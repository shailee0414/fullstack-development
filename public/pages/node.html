What is Node.js?

Node.js is an open-source, cross-platform JavaScript runtime environment built on Chrome's V8 engine that executes JavaScript outside the browser.

What Exactly is Node.js?

Node.js is NOT a language — it's a runtime environment that lets you run JavaScript on the server side. Before Node.js (2009), JavaScript could only run inside browsers.

Ryan Dahl created Node.js by embedding Google Chrome's V8 JavaScript engine (which compiles JS to machine code) into a C++ program that adds capabilities like file system access, networking, and process control — things browsers don't expose.

What Node.js Is NOT

  • Not a programming language — JavaScript is the language
  • Not a web framework — Express, Fastify are frameworks built ON Node
  • Not multithreaded (by default) — it runs on a single thread

Key Characteristics

  • Non-blocking I/O — doesn't wait for file/DB reads to finish
  • Event-driven — uses events and callbacks to handle async work
  • npm ecosystem — 2M+ packages available
  • Same language everywhere — JavaScript on frontend AND backend
💡 Node.js is perfect for I/O-intensive applications (APIs, chat apps, streaming). It struggles with CPU-intensive tasks (image processing, heavy math).

Runtime vs Language

JavaScript is the language. Node.js is the runtime — it provides the environment and APIs to execute that language outside a browser.

The Language vs Runtime Distinction

JavaScript defines the syntax, semantics, and standard library — Array, Object, Promise, JSON. This is standardized by ECMAScript.

A Runtime provides the environment to RUN JavaScript — it supplies the engine plus platform-specific APIs.

┌──────────────────────────────────────────────────────┐ │ YOUR JAVASCRIPT CODE │ ├──────────────────────────────────────────────────────┤ │ ECMAScript Standard (the language) │ │ Arrays, Objects, Promises, Map, Set, String, Math │ ├─────────────────────┬────────────────────────────────┤ │ BROWSER RUNTIME │ NODE.JS RUNTIME │ │ window, document │ fs, http, os, path, process │ │ localStorage │ require(), module.exports │ │ fetch, XMLHttp │ Buffer, Worker Threads │ │ navigator, alert │ child_process, cluster │ │ DOM / CSSOM │ EventEmitter, Stream │ ├─────────────────────┴────────────────────────────────┤ │ V8 Engine (compiles JS → machine code) │ └──────────────────────────────────────────────────────┘

What Does a Runtime Actually Do?

  • Provides the JS engine (V8) to parse and execute code
  • Provides platform APIs (file I/O, network, timers)
  • Manages memory (garbage collection via V8)
  • Provides the event loop mechanism
💡 This is why window.localStorage works in a browser but throws ReferenceError in Node — they share the same language but have different runtime APIs.

How Node.js Works — V8 Engine

V8 is Google's open-source JavaScript engine that compiles JS directly to machine code (JIT compilation) making it extremely fast.

What is V8?

V8 is a C++ program built by Google (used in Chrome). It takes your JavaScript source code and compiles it to native machine code that your CPU can execute directly — this is called JIT (Just-In-Time) compilation.

V8 Compilation Pipeline

Your JS Code ↓ Parser → AST (Abstract Syntax Tree) ↓ Ignition (Interpreter) → Bytecode ← runs immediately ↓ (if hot code - called repeatedly) TurboFan (Optimizing Compiler) → Optimized Machine Code ↓ CPU executes at near-native speed

Why JIT?

  • Interpreted: Easy to start, slow to run (Python approach)
  • Compiled: Slow to start, fast to run (C++ approach)
  • JIT: Best of both — starts fast, optimizes hot paths as they run

Node.js Architecture Stack

┌─────────────────────────────────┐ │ Your JS Code │ ├─────────────────────────────────┤ │ Node.js Standard Library │ (written in JS) ├─────────────────────────────────┤ │ Node.js Bindings (C++) │ bridges JS ↔ C++ ├──────────────┬──────────────────┤ │ V8 Engine │ libuv │ │ (executes │ (event loop, │ │ JS code) │ I/O, threads) │ ├──────────────┴──────────────────┤ │ Operating System │ └─────────────────────────────────┘

libuv is the C library that gives Node.js its non-blocking I/O capabilities. It manages the event loop, thread pool, and async I/O operations. Without libuv, Node would be just V8 with no async superpowers.

💡 libuv's thread pool (default size: 4) handles file I/O, DNS, and crypto operations that can't be done asynchronously at the OS level. This is why Node can do "non-blocking I/O" even when the OS operation is inherently synchronous.

Why Use Node.js?

Node.js excels at handling many concurrent I/O-bound connections with minimal overhead — making it ideal for real-time, API, and streaming applications.

Where Node.js Shines ✅

  • REST APIs & GraphQL servers — high throughput with low memory
  • Real-time apps — chat, notifications, live dashboards (WebSockets)
  • Streaming — video/audio streaming, file processing pipelines
  • Microservices — lightweight, fast startup
  • BFF (Backend for Frontend) — aggregating multiple API calls
  • CLI tools — build tools (webpack, vite), package managers

Where Node.js Struggles ❌

  • CPU-intensive work — image/video processing, ML, complex calculations
  • Heavy computation per request — blocks the single thread
  • Multi-threaded requirements — better handled by Go, Java, Rust

Node vs Traditional Thread-Per-Request Model

APACHE/JAVA (Thread-per-request): Request 1 → Thread 1 (waiting for DB...) — BLOCKED Request 2 → Thread 2 (waiting for DB...) — BLOCKED Request 3 → Thread 3 (waiting for DB...) — BLOCKED → 1000 requests = 1000 threads = high memory! NODE.JS (Event loop): Request 1 → sends DB query → returns to event loop (not blocked!) Request 2 → sends DB query → returns to event loop Request 3 → sends DB query → returns to event loop DB responds → event loop picks up, sends response → 1000 requests = 1 thread = low memory!
💡 LinkedIn reduced servers by 10x after switching to Node.js. Netflix reduced startup time by 70%. PayPal doubled requests per second while halving response time.

Single-Threaded Model

Node.js runs your JavaScript on a single main thread. "Single-threaded" means one piece of JS runs at a time — but I/O is delegated to OS/libuv threads behind the scenes.

What "Single-Threaded" Actually Means

Your JavaScript always runs on ONE thread — the main thread. Only one piece of JS code executes at any given moment. There is no parallel JS execution (unless you use Worker Threads).

BUT — the magic is that while JS code is "at rest" (waiting for I/O), the event loop can process OTHER callbacks. This is how Node achieves concurrency without multiple threads.

The Full Picture

Main Thread (your JS runs here): ┌──────────────────────────────────────────┐ │ JavaScript → Event Loop → Callbacks │ └──────────────────────────────────────────┘ ↓ delegates I/O to ↓ libuv Thread Pool (C++ threads, size 4): ┌──────────────────────────────────────────┐ │ Thread 1: reading file.txt │ │ Thread 2: DNS lookup │ │ Thread 3: crypto.pbkdf2() │ │ Thread 4: available │ └──────────────────────────────────────────┘ ↓ results posted back to ↓ Event Loop Queue → JS callback runs on main thread

Key Insight

When you call fs.readFile(), Node doesn't sit and wait. It hands the work to libuv, which uses OS-level APIs or its own thread pool. When the work is done, it posts the callback to the event loop queue, which runs it on the main JS thread.

⚠️ If you run a long synchronous operation (like a loop with 1 billion iterations), you WILL block the entire server — no other requests can be handled during that time.
single-threaded-demo.js
// Demonstrating single-threaded behavior

// ❌ THIS BLOCKS THE ENTIRE SERVER
app.get('/bad', (req, res) => {
  // This loop blocks the main thread for ~2 seconds
  // During this time, ALL other requests are frozen!
  let sum = 0;
  for (let i = 0; i < 5_000_000_000; i++) {
    sum += i;
  }
  res.json({ sum });
});

// ✅ Non-blocking — hands work off to background
app.get('/good', async (req, res) => {
  // This does NOT block — Node delegates to libuv thread pool
  const data = await fs.promises.readFile('bigfile.txt');
  // Other requests can be served while this is happening!
  res.json({ size: data.length });
});

// Proof of single thread — setInterval vs heavy sync work
setInterval(() => {
  console.log('tick');  // should print every 1 second
}, 1000);

// ❌ This blocks the interval above from firing
setTimeout(() => {
  const start = Date.now();
  while (Date.now() - start < 5000) {}  // blocks for 5 seconds!
  // 'tick' won't fire during these 5 seconds
}, 2000);

// Useful if you're doing lots of file I/O or crypto operations simultaneously

console.time('4-pbkdf2');

for (let i = 0; i < 4; i++) {

  console.timeEnd('4-pbkdf2');  // noticeably slower!

Non-Blocking I/O

Non-blocking I/O means Node initiates an I/O operation and immediately returns control — it doesn't sit and wait. The callback is invoked when the operation completes.

Blocking vs Non-Blocking

BLOCKING (synchronous): Time → [read file... waiting... waiting...] [process] [read next file... waiting...] [process] Thread: OCCUPIED ───────────────────────────────────────────────────────────────────────── NON-BLOCKING (asynchronous): Time → [start read] [start read] [start read] [process 1] [process 2] [process 3] Thread: FREE ──────────────────── FREE ──── processes callbacks as they arrive

How Node Achieves Non-Blocking I/O

  • Network I/O: Uses OS-level async mechanisms (epoll on Linux, kqueue on macOS, IOCP on Windows)
  • File I/O: libuv's thread pool does the work, posts result to event loop
  • DNS: libuv thread pool (getaddrinfo is blocking at OS level)
  • Crypto (pbkdf2, scrypt): libuv thread pool
  • Timers (setTimeout): libuv timer mechanism

The Callback Pattern

Node's original I/O API uses the error-first callback pattern: callback(err, result). First argument is always the error (or null), second is the result.

💡 The evolution: Callbacks → Promises → async/await. All three work in Node. Modern code uses async/await but you must understand callbacks for legacy codebases and deep internals.
non-blocking-io.js
const fs = require('fs');

// ❌ BLOCKING — synchronous (avoid in servers!)
try {
  const data = fs.readFileSync('./config.json');  // thread blocked here!
  const config = JSON.parse(data);
  console.log(config);
} catch (err) {
  console.error('Error:', err.message);
}

// ✅ NON-BLOCKING — callback style (original Node style)
fs.readFile('./config.json', 'utf8', (err, data) => {
  if (err) {
    console.error('Error:', err.message);
    return;
  }
  const config = JSON.parse(data);
  console.log(config);
});
console.log('This runs BEFORE the file is read!'); // prints first!

// ✅ NON-BLOCKING — Promise style
fs.promises.readFile('./config.json', 'utf8')
  .then(data => JSON.parse(data))
  .then(config => console.log(config))
  .catch(err => console.error(err));

// ✅ NON-BLOCKING — async/await (modern, cleanest)
async function loadConfig() {
  try {
    const data = await fs.promises.readFile('./config.json', 'utf8');
    return JSON.parse(data);
  } catch (err) {
    console.error('Failed to load config:', err.message);
    throw err;
  }
}

// Concurrent non-blocking I/O — this is where Node SHINES
async function processMultipleFiles() {
  console.time('total');
  
  // All 3 files read concurrently — NOT sequentially!
  const [config, users, products] = await Promise.all([
    fs.promises.readFile('./config.json', 'utf8'),
    fs.promises.readFile('./users.json', 'utf8'),
    fs.promises.readFile('./products.json', 'utf8'),
  ]);
  
  console.timeEnd('total'); // Takes time of slowest, not sum of all!
  return { config, users, products };
}

// Error-first callback pattern (convention)
function readUserFile(userId, callback) {
  fs.readFile(\`./users/\${userId}.json\`, 'utf8', (err, data) => {
    if (err) return callback(err);          // error first!
    callback(null, JSON.parse(data));       // null error, then data
  });
}

// Usage:

  if (err) { /* handle */ return; }
  console.log(user);
});

Event-Driven Architecture

Node.js is event-driven — instead of polling for I/O completion, it registers listeners and reacts when events fire. Everything in Node inherits from EventEmitter.

Observer Pattern at Scale

Event-driven programming is based on the Observer Pattern: you register interest in an event (.on('event', handler)), and when the event fires, all registered handlers are called.

Node's entire I/O model is built on this — the HTTP server "fires" an event when a new request arrives. Streams fire 'data' when chunks arrive. Processes fire 'exit' when terminating.

EventEmitter

The EventEmitter class is the backbone of Node's event system. Almost every Node.js core module (http, fs streams, net, child_process) extends EventEmitter.

EventEmitter instance: eventListeners = { 'data': [handler1, handler2], 'error': [handler3], 'end': [handler4], } .emit('data', chunk) → calls handler1(chunk) → calls handler2(chunk) (synchronously, in order)
💡 emit() is SYNCHRONOUS — all listeners run synchronously before emit() returns. This is important to understand for the event loop discussion.
event-driven.js
const { EventEmitter } = require('events');

// ─── Custom EventEmitter ───────────────────────────────────────

What is the Event Loop?

The Event Loop is the mechanism that allows Node.js to perform non-blocking I/O despite being single-threaded — by offloading operations and processing their callbacks when complete.

The Big Picture

The Event Loop is a C program (inside libuv) that continuously checks: "Is there any pending work? Are any callbacks ready to run?" It loops forever until there's nothing left to do.

┌─────────────────────────────────────────────────────────────┐ │ EVENT LOOP (libuv) │ │ │ │ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌────────┐ │ │ │ timers │→ │ pending │→ │ poll │→ │ check │ │ │ │setTimeout│ │ callbacks│ │ (I/O) │ │setImm- │ │ │ │setInterval│ │ │ │ │ │ediate │ │ │ └──────────┘ └──────────┘ └──────────┘ └────────┘ │ │ │ │ Between EACH phase: │ │ → process.nextTick() queue (all of them) │ │ → Promise microtask queue (all of them) │ └─────────────────────────────────────────────────────────────┘

How It Works Step by Step

  • 1. Your script runs: synchronous code executes on the call stack
  • 2. Async operations initiated: fs.readFile, setTimeout, etc. are registered with libuv
  • 3. Call stack empties: event loop takes over
  • 4. Event loop iterates: checks each phase for ready callbacks
  • 5. Callbacks execute: one at a time on the main thread
  • 6. Repeat: until no more callbacks pending

Priority Order

HIGHEST PRIORITY (runs between every event loop phase): 1. process.nextTick() callbacks 2. Promise .then() / .catch() (microtasks) THEN (event loop phases in order): 3. timers phase: setTimeout / setInterval (expired timers) 4. pending callbacks: I/O errors from previous iteration 5. idle, prepare: internal use 6. poll: I/O callbacks (fs, network) 7. check: setImmediate() callbacks 8. close callbacks: socket.on('close', ...) LOWEST PRIORITY
⚠️ This is THE #1 most asked Node.js interview question. Understand the priority queue order cold — nextTick > Promises > setTimeout > setImmediate.

Call Stack

The call stack is a LIFO (Last In, First Out) data structure that tracks which function is currently executing and where to return when it finishes.

How the Call Stack Works

When you call a function, a stack frame is pushed onto the stack containing the function's local variables and the return address. When the function returns, its frame is popped.

function main() { // 1. main() pushed to stack console.log('start'); // Stack: [main] greet('Shailesh'); // 2. greet() pushed console.log('end'); // Stack: [main, greet] } // 3. greet() returns, popped // Stack: [main] function greet(name) { // 4. main() returns, popped sayHello(name); // Stack: [] } function sayHello(name) { console.log('Hello', name); } // Call stack at deepest point: // [main → greet → sayHello]

Stack Overflow

If functions call each other recursively without a base case, the stack grows until it exceeds V8's limit (about 10,000-15,000 frames) → RangeError: Maximum call stack size exceeded

The Stack and Event Loop Relationship

The event loop only picks up callbacks from its queues when the call stack is empty. This is crucial — even if a setTimeout expires, its callback won't run until all synchronous code finishes and the stack empties.

Microtask Queue

The microtask queue holds Promise callbacks and queueMicrotask(). It is processed completely after each task, before the next macrotask runs.

What Goes in the Microtask Queue?

  • Promise .then() / .catch() / .finally() callbacks
  • queueMicrotask()
  • MutationObserver (browser)
  • Note: process.nextTick() is NOT technically a microtask but runs BEFORE the Promise microtask queue

Microtask Queue Rules

  • Processed completely between event loop phases
  • If a microtask adds another microtask → it also runs in current batch
  • This means infinite Promise chains = starvation of macrotasks (setTimeout, etc.)
Execution order in one "tick": 1. Run current synchronous task 2. Drain nextTick queue (ALL of them, including newly added ones) 3. Drain Promise microtask queue (ALL of them) 4. Move to next event loop phase / pick next macrotask 5. Repeat from step 2

Event Loop Phases

The event loop has 6 phases. Each phase has a FIFO queue of callbacks to execute. The loop iterates through these phases continuously.

The 6 Phases in Detail

┌───────────────────────────────────────────────────────────┐ │ PHASE 1: timers │ │ Runs setTimeout() and setInterval() callbacks whose │ │ delay has expired. Note: minimum delay, not exact! │ ├───────────────────────────────────────────────────────────┤ │ PHASE 2: pending callbacks │ │ TCP errors, deferred I/O callbacks from previous loop │ ├───────────────────────────────────────────────────────────┤ │ PHASE 3: idle, prepare │ │ Internal use only (you can ignore this) │ ├───────────────────────────────────────────────────────────┤ │ PHASE 4: poll ← WHERE NODE SPENDS MOST TIME │ │ Retrieves new I/O events. Executes I/O callbacks. │ │ If timers are ready → move to phase 1 │ │ If no I/O → waits here (blocking!) for new I/O events │ ├───────────────────────────────────────────────────────────┤ │ PHASE 5: check │ │ setImmediate() callbacks run here │ ├───────────────────────────────────────────────────────────┤ │ PHASE 6: close callbacks │ │ socket.on('close'), http.on('close'), etc. │ └───────────────────────────────────────────────────────────┘ Between EVERY phase transition: → Drain nextTick queue → Drain Promise microtask queue

The Poll Phase (Most Important)

The poll phase is where Node spends most of its time. It does two things:

  • Executes I/O event callbacks (file reads, network responses)
  • If nothing to do: blocks and waits for new I/O (prevents CPU spinning)

It leaves the poll phase when: a timer's threshold is met, or setImmediate() is scheduled.

process.nextTick()

process.nextTick() schedules a callback to run at the end of the current operation, before the event loop continues — even before Promises.

What Makes nextTick Special

process.nextTick() is technically NOT part of the event loop phases — it's processed by Node's C++ layer immediately after the current synchronous operation completes, before anything else (including Promise microtasks).

Despite its name, "next tick" doesn't mean "next event loop iteration" — it means "as soon as possible before moving anywhere else."

When to Use nextTick

  • Allow errors to be handled: If a constructor emits an error event, the caller hasn't had a chance to attach a listener yet. nextTick defers emission until after the constructor returns.
  • Guarantee async: Make an API consistently async even when result is already available
  • Before I/O begins: Do work after call stack unwinds but before any I/O
⚠️ Recursive nextTick calls will STARVE the event loop! Unlike setImmediate, nextTick processes ALL queued callbacks before moving on — including newly added ones.
process-nexttick.js
// ─── Real use case: ensure consistent async behavior ──────────
class DataLoader {
  constructor() {
    this.cache = null;
  }

  // BAD: sometimes sync, sometimes async → inconsistent!
  getData_bad(callback) {
    if (this.cache) {
      callback(null, this.cache); // ← synchronous! caller might not be ready
    } else {
      fetchFromDB((err, data) => {
        this.cache = data;
        callback(err, data);      // ← asynchronous
      });
    }
  }

  // GOOD: always async — nextTick makes sync path also async
  getData(callback) {
    if (this.cache) {
      process.nextTick(() => callback(null, this.cache)); // ← deferred
    } else {
      fetchFromDB((err, data) => {
        this.cache = data;
        callback(err, data);
      });
    }
  }
}

const loader = new DataLoader();
loader.getData((err, data) => {
  // With bad version: 'processing' might print AFTER this if cache exists!
  // With good version: ALWAYS prints after 'processing'
  console.log('got data');
});
console.log('processing'); // should always print before 'got data'

setTimeout vs setImmediate

Both schedule deferred execution. setImmediate runs in the check phase (after I/O). setTimeout runs in the timers phase after the specified delay.

┌────────────────────────────────────────────────────────────┐ │ setTimeout vs setImmediate │ ├────────────────┬───────────────────────────────────────────┤ │ │ setTimeout(fn, 0) │ setImmediate(fn) │ ├────────────────┼──────────────────────┼────────────────────┤ │ Phase │ timers (phase 1) │ check (phase 5) │ │ Minimum delay │ ~1-4ms (OS timer) │ no delay │ │ At top level │ non-deterministic │ non-deterministic │ │ Inside I/O cb │ AFTER setImmediate │ BEFORE setTimeout │ │ Use case │ delay execution │ after I/O, sooner │ └────────────────┴──────────────────────┴────────────────────┘

Key Rule for Interviews

At the top level (outside I/O): order is non-deterministic — depends on OS timer resolution.

Inside an I/O callback: setImmediate ALWAYS runs before setTimeout(0) — the check phase comes before the next timers phase in the loop.

    fn();
    setTimeout(run, delay); // schedule next AFTER fn() completes
  }, delay);

// recursive setTimeout: N ms AFTER fn completes

  return function(...args) {
    clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), delay);
  };
}

// Only logs: 'searching for: app'

Module System 🔥

CommonJS (require)

CommonJS is Node.js's original module system. Modules are loaded synchronously and cached after first load.

How CommonJS Works

Every file in Node.js is wrapped in a module wrapper function before execution:

(function(exports, require, module, __filename, __dirname) { // Your code here });

This gives each file its own scope (not global) and provides require, exports, and module.

Resolution Algorithm

  • Core module? → return it (e.g., require('fs'))
  • Starts with './' or '../'? → resolve as file path
  • Otherwise → search node_modules directories up the tree

Module Loading is Synchronous

require() is synchronous — it blocks until the module is loaded. This is fine at startup but problematic inside request handlers (don't require() inside loops or hot paths).

Module Caching

Node.js caches modules after the first require() — subsequent requires return the cached version. Modules are singletons.

Module Cache = Singleton Pattern

After first require(), the module's exports object is stored in require.cache keyed by the absolute file path. All subsequent requires return the same object — not a new one.

This means: if a module stores state (like a database connection, configuration object, or counter), all files sharing that require() get the SAME instance.

Practical Implications

  • DB connection pools: Create once, reuse everywhere — that's the cache working for you
  • Config objects: Loaded once, shared across app
  • Circular deps: Partially loaded cached module returned to break infinite loop
  • Testing: Must clear cache to re-test module initialization

Streams & Buffers

Streams process data piece by piece (chunks) without loading everything into memory. Essential for large files, video streaming, and upload handling.

Why Streams?

Without streams, reading a 2GB video file would load 2GB into RAM before sending it. With streams, you send chunks as they're read — memory usage stays at a few KB regardless of file size.

4 Types of Streams

  • Readable: source of data (fs.createReadStream, http.IncomingMessage, process.stdin)
  • Writable: destination (fs.createWriteStream, http.ServerResponse, process.stdout)
  • Duplex: both readable and writable (TCP sockets, net.Socket)
  • Transform: duplex that modifies data (zlib for compression, crypto streams)

Buffer

A Buffer is a fixed-size chunk of binary data in memory. It's how Node.js represents raw bytes — before streams were common, you'd read entire files into Buffers. Buffers are used internally by streams as chunks.

Reading 1GB file WITHOUT streams: [Read entire 1GB into RAM] → [Send to response] Memory usage: 1GB+ Reading 1GB file WITH streams: [Read chunk 1 64KB] → [Send chunk 1] → free memory [Read chunk 2 64KB] → [Send chunk 2] → free memory ... Memory usage: ~64KB constant!

Error Handling 🔥

Sync vs Async Error Handling

Sync errors are caught with try/catch. Async errors require different patterns: error-first callbacks, .catch() for Promises, or try/catch in async functions.

The Core Problem

try/catch only catches errors that happen synchronously in the same event loop tick. Errors in callbacks or async operations escape the try/catch.

try { // ← catch is set up here fs.readFile('file', callback); // ← this returns immediately! } catch(e) {} // ← will NOT catch callback errors! // The callback runs LATER in a different event loop tick // The try/catch is already gone by then

Error Types in Node

  • Synchronous errors: JSON.parse(), fs.readFileSync() — use try/catch
  • Callback errors: err-first pattern — check err argument
  • Promise rejections: .catch() or try/catch in async function
  • EventEmitter errors: 'error' event — MUST have listener or process crashes
  • Uncaught exceptions: process.on('uncaughtException')
  • Unhandled rejections: process.on('unhandledRejection')
⚠️ An EventEmitter that emits 'error' with no listener will crash the process. ALWAYS attach error listeners to streams, servers, and custom EventEmitters.

Global Error Handling

Handle errors that escape all local error boundaries — uncaughtException and unhandledRejection — to prevent silent crashes.

Global Error Events on process

  • uncaughtException: A synchronous error was thrown but not caught anywhere in the call stack → process would crash
  • unhandledRejection: A Promise was rejected but had no .catch() handler → in Node 15+, crashes the process
  • SIGTERM: Operating system signal to gracefully shut down (from Docker, Kubernetes, PM2)
  • SIGINT: Ctrl+C signal
⚠️ Using uncaughtException to "ignore" errors is dangerous — the process may be in an undefined state after an uncaught exception. Best practice: log the error, then restart the process (let PM2/Docker handle restart).

Middleware Deep Dive

Middleware are functions that have access to req, res, and next. They form a pipeline — each middleware can modify the request/response or pass control to the next one.

The Middleware Pipeline

HTTP Request ↓ [middleware 1] → next() → [middleware 2] → next() → [middleware 3] → next() → [Route handler] → res.send() ↓ HTTP Response If a middleware does NOT call next() and doesn't send a response: The request hangs forever!

Middleware Function Signature

(req, res, next) => {} — Three parameters define a regular middleware.

(err, req, res, next) => {} — FOUR parameters define error-handling middleware (Express detects this automatically).

What Middleware Can Do

  • Execute any code (logging, validation)
  • Modify req object (attach user, parse body)
  • Modify res object (set headers)
  • End the request-response cycle (return error, send response)
  • Call next() to pass to next middleware
  • Call next(err) to skip to error handler

REST API Design

REST (Representational State Transfer) is an architectural style for designing APIs using HTTP methods, status codes, and resource-based URLs.

REST Principles

  • Stateless: Each request contains all info needed. No session on server.
  • Resource-based URLs: Nouns, not verbs. /users not /getUsers
  • HTTP methods as verbs: GET (read), POST (create), PUT (replace), PATCH (update), DELETE (remove)
  • Consistent status codes: 200 OK, 201 Created, 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 500 Server Error

HTTP Status Code Reference

2xx Success: 200 OK → GET, PUT, PATCH success 201 Created → POST success (include Location header) 204 No Content → DELETE success 4xx Client Error: 400 Bad Request → invalid data, missing fields 401 Unauthorized → not authenticated (no/bad token) 403 Forbidden → authenticated but not authorized 404 Not Found → resource doesn't exist 409 Conflict → duplicate resource (email already used) 422 Unprocessable → validation failed 429 Too Many Req → rate limit exceeded 5xx Server Error: 500 Internal Error → unexpected server error 502 Bad Gateway → upstream service failure 503 Service Unavail → server down for maintenance

JWT (JSON Web Token)

JWT is a compact, self-contained token format for transmitting claims between parties. Stateless authentication — server doesn't store sessions.

JWT Structure

JWT Flow

1. User logs in with email + password 2. Server verifies credentials 3. Server creates JWT with userId, role, expiry 4. Server signs JWT with secret key → sends to client 5. Client stores JWT (localStorage or httpOnly cookie) 6. Client sends JWT in Authorization: Bearer {token} header 7. Server verifies signature, extracts userId/role → no DB lookup needed! 8. Server handles request with user context

Access Token + Refresh Token Pattern

  • Access token: Short-lived (15min), sent with every request
  • Refresh token: Long-lived (7-30 days), stored in httpOnly cookie, used ONLY to get new access tokens
  • If access token stolen: expires in 15 min (limited damage)
  • If refresh token stolen: can be revoked in DB
⚠️ JWTs cannot be invalidated before expiry (stateless!). To "log out" or "revoke", use a token blacklist in Redis. This is JWT's main weakness vs sessions.

const jwt = require('jsonwebtoken');
const bcrypt = require('bcryptjs');

// ─── Token generation ─────────────────────────────────────────
function generateTokens(userId, role) {
  const accessToken = jwt.sign(
    { userId, role },           // payload (public claims)
    process.env.JWT_SECRET,     // secret key
    { expiresIn: '15m' }        // short lived
  );

  const refreshToken = jwt.sign(
    { userId },
    process.env.REFRESH_SECRET, // different secret!
    { expiresIn: '7d' }
  );

  return { accessToken, refreshToken };
}

app.post('/auth/login', async (req, res, next) => {
  try {
    const { email, password } = req.body;

    const user = await User.findOne({ email }).select('+password');
    if (!user) return res.status(401).json({ error: 'Invalid credentials' });

    const isValid = await bcrypt.compare(password, user.password);
    if (!isValid) return res.status(401).json({ error: 'Invalid credentials' });

    const { accessToken, refreshToken } = generateTokens(user._id, user.role);

    // Store refresh token in DB (for revocation)
    await RefreshToken.create({ token: refreshToken, userId: user._id });

    // Send refresh token as httpOnly cookie (XSS-safe)
    res.cookie('refreshToken', refreshToken, {
      httpOnly: true,   // can't be accessed by JS
      secure: true,     // HTTPS only
      sameSite: 'strict',
      maxAge: 7 * 24 * 60 * 60 * 1000, // 7 days in ms
    });

    res.json({ accessToken }); // access token in response body
  } catch (err) { next(err); }
});

// ─── Auth middleware ──────────────────────────────────────────
async function authenticate(req, res, next) {
  try {
    const authHeader = req.headers.authorization;
    if (!authHeader?.startsWith('Bearer ')) {
      return res.status(401).json({ error: 'No token provided' });
    }

    const token = authHeader.split(' ')[1];
    const decoded = jwt.verify(token, process.env.JWT_SECRET);

    // Optionally: check if token is blacklisted
    const isBlacklisted = await redis.get(\`blacklist:\${token}\`);
    if (isBlacklisted) return res.status(401).json({ error: 'Token revoked' });

    req.user = decoded; // { userId, role, iat, exp }
    next();
  } catch (err) {
    if (err.name === 'TokenExpiredError') {
      return res.status(401).json({ error: 'Token expired', code: 'TOKEN_EXPIRED' });
    }
    if (err.name === 'JsonWebTokenError') {
      return res.status(401).json({ error: 'Invalid token' });
    }
    next(err);
  }
}

  try {
    const { refreshToken } = req.cookies;
    if (!refreshToken) return res.status(401).json({ error: 'No refresh token' });

    const decoded = jwt.verify(refreshToken, process.env.REFRESH_SECRET);

    // Verify it's in our DB (not revoked)
    const stored = await RefreshToken.findOne({ token: refreshToken });
    if (!stored) return res.status(401).json({ error: 'Refresh token revoked' });

    const user = await User.findById(decoded.userId);
    const { accessToken } = generateTokens(user._id, user.role);

    res.json({ accessToken });

  // Delete refresh token from DB
  await RefreshToken.deleteOne({ token: req.cookies.refreshToken });
  res.clearCookie('refreshToken');
  res.json({ message: 'Logged out successfully' });
});

CORS

Cross-Origin Resource Sharing (CORS) is a browser security mechanism that controls which origins can access your API.

Same-Origin Policy

Browsers enforce that a page at https://app.com cannot make fetch requests to https://api.otherdomain.com unless that other domain explicitly allows it via CORS headers.

Same origin = same protocol + same host + same port. Any difference = cross-origin.

Preflight Request

For "non-simple" requests (custom headers, non-GET/POST, JSON body), the browser sends an OPTIONS preflight request first to ask "do you allow this?" The server must respond with appropriate CORS headers before the actual request is sent.

Browser Server | | |-- OPTIONS /api/users -------→ | | Origin: https://app.com | | Access-Control-Request-Method: POST | | | |← 204 No Content ---------------| | Access-Control-Allow-Origin: https://app.com | Access-Control-Allow-Methods: POST | | |-- POST /api/users ----------→ | (actual request) |← 200 OK -----------------------|

Key CORS Headers

  • Access-Control-Allow-Origin: which origins allowed (* = all)
  • Access-Control-Allow-Methods: GET, POST, PUT, DELETE, etc.
  • Access-Control-Allow-Headers: Content-Type, Authorization, etc.
  • Access-Control-Allow-Credentials: true (for cookies/auth headers)
  • Access-Control-Max-Age: how long to cache preflight (seconds)

const allowedOrigins = [
  'https://app.policybazaar.com',
  'https://www.policybazaar.com',
  'https://admin.policybazaar.com',
  ...(process.env.NODE_ENV === 'development' ? ['http://localhost:3000'] : []),
];

const corsOptions = {
  origin: (origin, callback) => {
    // Allow requests with no origin (server-to-server, Postman)
    if (!origin) return callback(null, true);
    
    if (allowedOrigins.includes(origin)) {
      callback(null, true);
    } else {
      callback(new Error(\`CORS: Origin \${origin} not allowed\`));
    }
  },
  methods: ['GET', 'POST', 'PUT', 'PATCH', 'DELETE', 'OPTIONS'],
  allowedHeaders: ['Content-Type', 'Authorization', 'X-Request-ID'],
  credentials: true,          // allow cookies
  maxAge: 86400,              // cache preflight for 24 hours
  optionsSuccessStatus: 204,  // some legacy browsers need 200
};

app.use(cors(corsOptions));

// ─── Manual CORS (without package, to understand internals) ──
app.use((req, res, next) => {
  const origin = req.headers.origin;
  
  if (allowedOrigins.includes(origin)) {
    res.setHeader('Access-Control-Allow-Origin', origin);
    res.setHeader('Vary', 'Origin'); // tell CDN to cache per origin
  }
  
  res.setHeader('Access-Control-Allow-Methods', 'GET, POST, PUT, DELETE');
  res.setHeader('Access-Control-Allow-Headers', 'Content-Type, Authorization');
  res.setHeader('Access-Control-Allow-Credentials', 'true');
  
  // Handle preflight
  if (req.method === 'OPTIONS') {
    res.setHeader('Access-Control-Max-Age', '86400');
    return res.status(204).end();
  }
  
  next();
});

What is Redis?

Redis is an in-memory data store used as a cache, message broker, and session store. Sub-millisecond response times make it ideal for caching hot data.

Why Redis?

MongoDB query: ~10-50ms. Redis GET: ~0.1ms. For frequently-read data, caching in Redis reduces DB load by 90%+ and speeds up responses dramatically.

Redis Data Structures

  • String: key-value, numbers — cache JSON, counters
  • Hash: field-value map — store objects without serializing
  • List: linked list — queues, recent activity feeds
  • Set: unique values — unique visitors, tags
  • Sorted Set: scored members — leaderboards, rate limiting
  • Pub/Sub: messaging — real-time notifications
  • Stream: append-only log — event sourcing
Without Redis cache: User → Node.js → MongoDB (50ms) → User [every request!] With Redis cache: User → Node.js → Redis (0.1ms) → User [cache hit, 99% of requests] User → Node.js → MongoDB (50ms) → Redis (store) → User [cache miss, 1%]

function cacheMiddleware(ttlSeconds = 300) {
  return async (req, res, next) => {
    const key = \`cache:\${req.method}:\${req.url}:\${req.user?.id || 'anon'}\`;
    
    const cached = await redis.get(key);
    if (cached) {
      console.log('Cache HIT:', key);
      return res.json(JSON.parse(cached));
    }
    
    // Capture the response to cache it
    const originalJson = res.json.bind(res);
    res.json = async (data) => {
      await redis.set(key, JSON.stringify(data), { EX: ttlSeconds });
      originalJson(data);
    };
    
    next();
  };
}

// ─── Rate limiting with Redis ─────────────────────────────────
async function rateLimit(identifier, maxRequests, windowSeconds) {
  const key = \`rl:\${identifier}\`;
  
  const current = await redis.incr(key);
  if (current === 1) {
    await redis.expire(key, windowSeconds); // set TTL on first hit
  }
  
  return {
    allowed: current <= maxRequests,
    count: current,
    remaining: Math.max(0, maxRequests - current),
  };
}

  const identifier = req.user?.id || req.ip;
  const { allowed, remaining } = await rateLimit(identifier, 100, 60);
  
  res.setHeader('X-RateLimit-Remaining', remaining);
  
  if (!allowed) {
    return res.status(429).json({ error: 'Rate limit exceeded. Try again later.' });
  }
  next();
});

Worker Threads

Worker Threads enable true multi-threading in Node.js for CPU-intensive tasks — without blocking the main event loop.

Why Worker Threads?

The event loop is single-threaded. A CPU-heavy task (image resize, PDF generation, complex calculation) blocks it for all other requests. Worker Threads solve this by running CPU tasks in separate V8 instances with their own event loop.

WITHOUT Worker Threads: Main thread: [req1] [CPU task 5000ms] [req2] [req3] ← all requests wait during CPU task! → WITH Worker Threads: Main thread: [req1] ─────────────────────→ [res1] [req2] ─────────────────────→ [res2] Worker 1: [CPU task 5000ms] ← req1, req2 handled while CPU task runs! →

Worker Threads vs Cluster vs Child Process

  • Worker Threads: CPU tasks within same process, shared memory available
  • Cluster: Multiple Node processes for I/O scaling on multi-core CPUs
  • Child Process: Run external scripts, isolated process, pipe communication
worker-threads.js
const { Worker, isMainThread, parentPort, workerData } = require('worker_threads');

// ─── worker.js (the worker file) ─────────────────────────────
if (!isMainThread) {
  // This code runs in the worker thread
  const { data, type } = workerData;

  let result;
  if (type === 'fibonacci') {
    function fib(n) {
      if (n <= 1) return n;
      return fib(n - 1) + fib(n - 2);
    }
    result = fib(data);
  } else if (type === 'sort') {
    result = data.sort((a, b) => a - b);
  }

  // Send result back to main thread
  parentPort.postMessage({ result, type });
}

// ─── main.js ─────────────────────────────────────────────────
function runInWorker(data, type) {
  return new Promise((resolve, reject) => {
    const worker = new Worker(__filename, {
      workerData: { data, type },
    });

    worker.on('message', resolve);
    worker.on('error', reject);
    worker.on('exit', (code) => {
      if (code !== 0) reject(new Error(\`Worker exited with code \${code}\`));
    });
  });
}

// API endpoint that uses worker for CPU task
app.get('/fibonacci/:n', async (req, res, next) => {
  try {
    const n = parseInt(req.params.n);
    if (n > 50) return res.status(400).json({ error: 'Too large' });

    // Runs in worker thread — doesn't block event loop!
    const { result } = await runInWorker(n, 'fibonacci');
    res.json({ n, fibonacci: result });
  } catch (err) { next(err); }
});

Cluster Module

The cluster module forks Node.js into multiple processes (one per CPU core) to handle more concurrent requests and utilize multi-core machines.

The Problem

A single Node.js process runs on ONE CPU core. An 8-core machine is 87.5% idle! Cluster forks worker processes that all share the same port, and the OS distributes connections among them.

Master Process (cluster.isMaster) ├── Worker Process 1 (CPU core 1) → handles requests ├── Worker Process 2 (CPU core 2) → handles requests ├── Worker Process 3 (CPU core 3) → handles requests ├── Worker Process 4 (CPU core 4) → handles requests ├── Worker Process 5 (CPU core 5) → handles requests ├── Worker Process 6 (CPU core 6) → handles requests ├── Worker Process 7 (CPU core 7) → handles requests └── Worker Process 8 (CPU core 8) → handles requests All workers share port 3000 — OS load balances between them

Cluster vs PM2

In production, you typically use PM2 (pm2 start app.js -i max) instead of writing your own cluster code — PM2 handles clustering, restarts, and monitoring automatically.

cluster.js
const cluster = require('cluster');
const os = require('os');
const http = require('http');

const NUM_CPUS = os.cpus().length; // e.g., 8

if (cluster.isMaster) {
  console.log(\`Master PID \${process.pid} starting \${NUM_CPUS} workers\`);

  // Fork a worker for each CPU core
  for (let i = 0; i < NUM_CPUS; i++) {
    cluster.fork();
  }

  // Restart workers that die
  cluster.on('exit', (worker, code, signal) => {
    console.warn(\`Worker \${worker.process.pid} died (signal: \${signal}). Restarting...\`);
    cluster.fork(); // replace the dead worker
  });

  // Send message to all workers
  cluster.on('online', (worker) => {
    console.log(\`Worker \${worker.process.pid} is online\`);
    worker.send({ type: 'config', data: { maxConnections: 1000 } });
  });

} else {
  // Workers run the actual server
  const express = require('express');
  const app = express();

  app.get('/', (req, res) => {
    res.send(\`Response from worker \${process.pid}\`);
  });

  // CPU-intensive route (each worker handles it independently)
  app.get('/heavy', (req, res) => {
    let result = 0;
    for (let i = 0; i < 1e8; i++) result += i; // heavy computation
    res.json({ result, pid: process.pid });
  });

  // Receive messages from master
  process.on('message', (msg) => {
    if (msg.type === 'config') {
      console.log(\`Worker \${process.pid} configured:\`, msg.data);
    }
  });

  app.listen(3000, () => {
    console.log(\`Worker \${process.pid} listening on port 3000\`);
  });
}

WebSockets

WebSockets provide full-duplex, persistent TCP connections between client and server — enabling real-time bidirectional communication without HTTP overhead.

HTTP vs WebSocket

HTTP (request-response): Client → [request] → Server Client ← [response] ← Server Connection closed. New connection for every message. WebSocket (persistent): Client ←──────────── persistent connection ──────────────→ Server [msg] → [msg] ← [msg] → [msg] ← [msg] ← [msg] One connection, data flows freely in both directions!

WebSocket Handshake

WebSocket starts as an HTTP request with Upgrade: websocket header. The server responds with 101 Switching Protocols — then they switch to WebSocket protocol. After that, no more HTTP.

Use Cases

  • Chat applications
  • Live notifications
  • Real-time dashboards
  • Collaborative editing (Google Docs style)
  • Live sports scores / stock prices
  • Multiplayer games

Docker & Containerization

Docker packages your Node.js app and all its dependencies into a container that runs identically everywhere — dev, staging, production.

Why Docker?

"It works on my machine" → Docker makes it work everywhere. A container includes Node.js version, npm packages, OS-level dependencies — everything is bundled.

Key Concepts

  • Image: Blueprint — built from Dockerfile, immutable
  • Container: Running instance of an image
  • Dockerfile: Instructions to build an image
  • docker-compose: Define multi-container apps (Node + MongoDB + Redis)
  • Layer caching: Each Dockerfile instruction is a cached layer — order matters for build speed

Multi-Stage Builds

Build the app in one stage (with dev dependencies), copy only the production artifact to a smaller final image. This reduces image size from ~1GB to ~100MB.

Dockerfile + docker-compose.yml

Testing with Jest & Supertest

Jest for unit testing, Supertest for integration testing HTTP endpoints without spinning up a real server.

Testing Pyramid

▲ /E2E\ few — slow, expensive, brittle /──────\ / Integ \ some — test API endpoints with DB /──────────\ / Unit \ many — fast, isolated, mocked deps ──────────────────

Types in Node.js Testing

  • Unit tests: Test a function in isolation. Mock all external dependencies (DB, Redis, external APIs). Fast.
  • Integration tests: Test an API endpoint with real (or in-memory) DB. Supertest + Jest.
  • E2E tests: Full app testing. Usually with Playwright/Cypress on the frontend.

Jest Key Concepts

  • jest.fn(): Create a mock function that tracks calls
  • jest.mock(): Auto-mock an entire module
  • jest.spyOn(): Mock a specific method on an object
  • beforeEach/afterEach: Setup/teardown per test
  • beforeAll/afterAll: Setup/teardown per test suite
