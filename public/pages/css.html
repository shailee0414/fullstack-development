Cascade & Specificity

Core

The Cascade is the algorithm CSS uses to decide which rule "wins" when multiple rules target the same element. It has four stages evaluated in order: Origin & Importance → Specificity → Order of Appearance.

1. Origin Priority (high → low)

Inline !important > User agent !important > Author !important > Inline styles > Author stylesheets > User preferences > Browser defaults

2. Specificity Score

Calculated as (a, b, c) — a = IDs, b = classes/attrs/pseudo-classes, c = elements/pseudo-elements. Higher score wins.

SelectorSpecificityScore (a,b,c)
*Universal — zero specificity(0,0,0)
divElement(0,0,1)
.classClass(0,1,0)
[type="text"]Attribute(0,1,0)
:hoverPseudo-class(0,1,0)
::beforePseudo-element(0,0,1)
#idID(1,0,0)
style=""Inline style(1,0,0,0) beats all
!importantOverrides everythingNuclear option ☢
specificity-examples.css
/* Specificity: (0,0,1) */
p { color: blue; }

/* Specificity: (0,1,0) — wins over element */
.text { color: red; }

/* Specificity: (0,1,1) */
p.text { color: green; }

/* Specificity: (1,0,0) — ID wins */
#main { color: purple; }

/* Specificity: (1,1,1) — ID + class + element */
#main .text p { color: orange; }

/* :is() and :not() take specificity of their MOST specific argument */
:is(#id, .class) { color: pink; }  /* specificity: (1,0,0) */

/* :where() always has 0 specificity — great for resets */
:where(#id, .class) { color: gray; }  /* specificity: (0,0,0) */
Avoid !important: It breaks the natural cascade and makes debugging very difficult. Use it only for utility classes or overriding third-party styles you can't change.

Box Model

Critical

Every HTML element is a rectangular box made of four layers from inside out: Content → Padding → Border → Margin. Understanding this is fundamental to all CSS layout.

box-model.css
.box {
  width: 200px;          /* content area width */
  height: 100px;

  padding: 20px;         /* space INSIDE border */
  border: 2px solid blue;
  margin: 16px;          /* space OUTSIDE border */

  /* content-box (default): total = 200+40+4 = 244px wide */
  box-sizing: content-box;

  /* border-box: width INCLUDES padding + border */
  /* total = exactly 200px. Much more intuitive! */
  box-sizing: border-box;
}

/* Universal border-box reset — use in EVERY project */
*, *::before, *::after {
  box-sizing: border-box;
}

/* Margin collapsing — adjacent margins merge */
/* p has margin-bottom:16px, next p has margin-top:16px */
/* Gap between them = 16px (not 32px) — they COLLAPSE */
p { margin: 16px 0; }

/* outline — like border but doesn't affect layout/box model */
.focused { outline: 2px solid blue; outline-offset: 2px; }
Margin Collapsing Rules: Vertical margins between block elements collapse to the larger of the two values. Margins collapse between parent and first/last child if no border/padding/BFC separates them. Horizontal margins never collapse.

positioning.css
/* relative — offset from where it would normally be */
.badge {
  position: relative;
  top: -4px;   /* moves up 4px (still in flow) */
}

/* absolute — positioned in nearest non-static ancestor */
.parent { position: relative; }
.tooltip {
  position: absolute;
  top: 100%;    /* just below parent */
  left: 50%;
  transform: translateX(-50%); /* center horizontally */
}

/* fixed — always in viewport */
.navbar {
  position: fixed;
  top: 0; left: 0; right: 0;
}

/* sticky — scrolls normally, then "sticks" */
.section-header {
  position: sticky;
  top: 60px;  /* sticks 60px from viewport top */
  z-index: 10;
}

/* Centering with absolute — classic technique */
.centered {
  position: absolute;
  top: 50%; left: 50%;
  transform: translate(-50%, -50%);
}
Sticky gotcha: position: sticky requires a defined top/bottom/left/right value to work. It also won't work if any ancestor has overflow: hidden or overflow: auto — this is a very common bug.

Flexbox — Deep Dive

Layout

Flexbox is a one-dimensional layout system. You define a main axis (row or column) and items are placed along it. The container controls alignment; children control their own growth/shrink. It excels at distributing space among items in a single direction.

flexbox-container.css
.flex-container {
  display: flex;

  /* MAIN AXIS DIRECTION */
  flex-direction: row;           /* row | row-reverse | column | column-reverse */

  /* WRAPPING */
  flex-wrap: nowrap;            /* nowrap | wrap | wrap-reverse */
  flex-flow: row wrap;          /* shorthand: direction + wrap */

  /* MAIN AXIS ALIGNMENT (horizontal when row) */
  justify-content: flex-start;  /* flex-start | flex-end | center */
                                   /* space-between | space-around | space-evenly */

  /* CROSS AXIS ALIGNMENT (vertical when row) */
  align-items: stretch;         /* stretch | flex-start | flex-end | center | baseline */

  /* MULTI-LINE CROSS AXIS (only when flex-wrap: wrap) */
  align-content: flex-start;    /* same values as justify-content */

  /* GAP between items */
  gap: 16px;                     /* row-gap + column-gap */
  gap: 16px 24px;                /* row-gap column-gap */
}
flexbox-children.css
.flex-item {
  /* flex-grow: how much free space to absorb (default 0) */
  flex-grow: 1;    /* item grows to fill available space */

  /* flex-shrink: how much to shrink when overflowing (default 1) */
  flex-shrink: 0;  /* item won't shrink */

  /* flex-basis: initial size before growing/shrinking */
  flex-basis: 200px;  /* or auto, 0, 30%, etc. */

  /* Shorthand: flex: grow shrink basis */
  flex: 1 1 auto;   /* = flex: 1 */
  flex: 0 0 200px;  /* fixed size, no grow/shrink */

  /* Override align-items for individual item */
  align-self: flex-end;

  /* Order — default 0, negative moves earlier */
  order: -1;  /* moves to start */
}

/* Common patterns */
/* Center anything: */
.center { display: flex; align-items: center; justify-content: center; }

/* Sticky footer: */
body { display: flex; flex-direction: column; min-height: 100vh; }
main { flex: 1; }  /* grows to push footer down */

/* Space between nav items, one item pushed right: */
.last-item { margin-left: auto; }

CSS Grid — Deep Dive

Layout

CSS Grid is a two-dimensional layout system — it handles both rows and columns simultaneously. While Flexbox is best for one direction, Grid excels when you need to control placement across both axes at once.

css-grid-container.css
.grid {
  display: grid;

  /* Define columns */
  grid-template-columns: 200px 1fr 1fr;     /* fixed + 2 equal fractions */
  grid-template-columns: repeat(3, 1fr);   /* 3 equal columns */
  grid-template-columns: repeat(auto-fill, minmax(200px, 1fr));  /* responsive! */

  /* Define rows */
  grid-template-rows: auto 1fr auto;        /* header, content, footer */

  /* Named areas — visual layout! */
  grid-template-areas:
    "header header header"
    "sidebar main main"
    "footer footer footer";

  gap: 24px;            /* gap between tracks */
  row-gap: 16px;
  column-gap: 24px;

  /* Align all items in cells */
  justify-items: start;   /* inline axis: start|end|center|stretch */
  align-items: stretch;   /* block axis */

  /* Align the grid itself in its container */
  justify-content: center;
  align-content: start;

  /* Implicit tracks — auto-created rows when items overflow */
  grid-auto-rows: 200px;
  grid-auto-flow: dense;  /* fill gaps automatically */
}
css-grid-items.css
/* Place items by line number */
.header {
  grid-column: 1 / -1;    /* span full width (1 to last line) */
  grid-row: 1;
}

/* Place by area name */
.sidebar { grid-area: sidebar; }
.main    { grid-area: main; }

/* Span multiple tracks */
.wide-card {
  grid-column: span 2;   /* spans 2 columns */
  grid-row: span 3;      /* spans 3 rows */
}

/* Override container alignment for one item */
.special {
  justify-self: center;
  align-self: end;
}

/* Holy Grail Layout in 10 lines */
.page {
  display: grid;
  grid-template:
    "header" 60px
    "nav main aside" 1fr
    "footer" 40px
    / 200px 1fr 160px;
  min-height: 100vh;
}
auto-fill vs auto-fit: Both create as many columns as fit. With auto-fill, empty tracks are preserved (grid stays the same width). With auto-fit, empty tracks collapse and items stretch to fill the space. Use auto-fit when you want items to grow and fill the container.

Responsive Design

Layout

Mobile-first means writing your base styles for small screens, then using min-width media queries to add styles as the screen gets larger. This is the preferred modern approach because it starts with constraints and progressively enhances.

responsive.css
/* Mobile first — base styles are for mobile */
.container {
  width: 100%;
  padding: 0 16px;
}

/* Tablet and up */
@media (min-width: 768px) {
  .container { max-width: 720px; margin: 0 auto; }
  .grid { grid-template-columns: repeat(2, 1fr); }
}

/* Desktop */
@media (min-width: 1024px) {
  .container { max-width: 960px; }
  .grid { grid-template-columns: repeat(3, 1fr); }
}

/* Common breakpoints */
/* sm:  640px  — small phones landscape */
/* md:  768px  — tablets */
/* lg:  1024px — laptops */
/* xl:  1280px — large screens */
/* 2xl: 1536px — very large screens */

/* Intrinsically responsive grid — no media queries needed! */
.auto-grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(min(280px, 100%), 1fr));
  gap: 24px;
}

Media Queries

Layout
media-queries.css
/* Width-based */
@media (min-width: 768px) { ... }
@media (max-width: 767px) { ... }
@media (768px <= width <= 1024px) { ... }  /* range syntax (modern) */

/* Orientation */
@media (orientation: landscape) { ... }
@media (orientation: portrait) { ... }

/* Print styles */
@media print {
  nav, .ads, button { display: none; }
  body { font-size: 12pt; color: black; }
}

/* Dark mode preference */
@media (prefers-color-scheme: dark) {
  :root { --bg: #111; --text: #eee; }
}

/* Reduced motion — VERY important for accessibility */
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}

/* Hover capability (true pointer device) */
@media (hover: hover) { ... }  /* only on devices with real hover */

/* High resolution (Retina) */
@media (min-resolution: 2dppx) {
  .logo { background-image: url('logo@2x.png'); }
}

/* Logical AND: comma = OR, and = AND */
@media (min-width: 768px) and (orientation: landscape) { ... }
@media (max-width: 480px), (min-width: 900px) { ... }  /* OR */
@media not (max-width: 767px) { ... }  /* NOT */

Container Queries

Modern CSS

Container Queries are a game-changer. Instead of responding to the viewport size, components respond to the size of their own container. A card can be wide in a sidebar and narrow in a grid — same component, different layout.

container-queries.css
/* 1. Define a containment context on the parent */
.card-wrapper {
  container-type: inline-size;  /* or: size, normal */
  container-name: card;          /* optional name */
}

/* 2. Query the container, not the viewport */
@container (min-width: 400px) {
  .card {
    display: flex;
    flex-direction: row;   /* horizontal when container is wide */
  }
}

/* 3. Target a named container */
@container card (min-width: 600px) {
  .card__image { width: 40%; }
}

/* Container query units — relative to container size */
.card h2 { font-size: 5cqi; }  /* 5% of container inline size */
/* cqi = container inline size */
/* cqb = container block size */
/* cqw = container width */
/* cqh = container height */

CSS Animations

Visual
animations.css
/* 1. Define keyframes */
@keyframes slideIn {
  from { transform: translateY(-20px); opacity: 0; }
  to   { transform: translateY(0); opacity: 1; }
}

@keyframes pulse {
  0%   { transform: scale(1); }
  50%  { transform: scale(1.1); }
  100% { transform: scale(1); }
}

@keyframes spin {
  to { transform: rotate(360deg); }
}

/* 2. Apply animation */
.alert {
  /* name | duration | timing | delay | iteration | direction | fill-mode | play-state */
  animation: slideIn 400ms ease-out 0s 1 normal both running;

  /* Shorthand examples */
  animation: slideIn 400ms ease-out;
  animation: pulse 1s ease-in-out infinite;
}

/* Multiple animations */
.loader { animation: spin 1s linear infinite, pulse 2s ease infinite; }

/* fill-mode values */
/* none     — element returns to initial state before/after */
/* forwards — keeps last keyframe state after ending */
/* backwards— applies first keyframe during delay */
/* both     — forwards + backwards */

/* Pause/resume animation */
.paused { animation-play-state: paused; }

/* Staggered animations with CSS custom properties */
.item { animation: fadeUp 0.4s ease var(--delay) both; }

SCSS Introduction

SCSS

SCSS (Sassy CSS) is a CSS preprocessor — it adds programming features to CSS and compiles to regular CSS. There are two syntaxes: SCSS (uses {} and ; — looks like CSS) and Sass (indented, no braces). SCSS is more popular because it's a superset of CSS — any valid CSS is valid SCSS.

SCSS Advantages

Variables, nesting, mixins, functions, loops, conditionals, partials, modules. DRY code. Better organization. Compile-time errors. Everything compiles to plain CSS.

CSS Variables vs SCSS Variables

SCSS variables ($var) are compile-time — they disappear after compilation. CSS variables (--var) are runtime — they live in the browser, can be changed with JS, respond to media queries. Modern preference: use both!

Stacking Context

Tricky

A stacking context is a 3D layer — elements inside it are painted together and their z-index values only compete within that context. This is why a child with z-index: 9999 can still be below another element if its parent has a lower stacking order.

What creates a new stacking context:

position + z-index (not auto) opacity < 1 transform filter will-change isolation: isolate mix-blend-mode contain: layout
stacking.css
/* Problem: child's z-index is trapped in parent's context */
.parent-a {
  position: relative;
  z-index: 1;             /* creates stacking context */
  opacity: 0.9;           /* ALSO creates stacking context */
}
.child {
  position: relative;
  z-index: 9999;          /* stuck inside parent's context */
  /* will NEVER appear above parent-b if parent-b z-index > 1 */
}
.parent-b {
  position: relative;
  z-index: 2;             /* above parent-a entirely */
}

/* Solution: isolation:isolate — creates stacking context cleanly */
.safe-layer { isolation: isolate; }

/* Paint order within same stacking context (back to front): */
/* 1. Root element background/borders */
/* 2. Negative z-index stacking contexts */
/* 3. Block-level boxes */
/* 4. Float boxes */
/* 5. Inline boxes */
/* 6. z-index: 0 / auto */
/* 7. Positive z-index */

Block Formatting Context (BFC)

Tricky

A Block Formatting Context (BFC) is an isolated layout environment — its children don't affect outside elements and vice versa. BFCs solve several classic CSS problems: float containment, margin collapse prevention, and preventing overlap with floats.

What creates a BFC:

overflow (not visible) display: flow-root display: flex/grid position: absolute/fixed float (any) contain: layout/content

/* Classic clearfix — OLD way to contain floats */
.clearfix::after {

  clear: both;

/* MODERN way — display: flow-root creates a BFC */
.float-container { display: flow-root; }

/* BFC prevents margin collapse */
.parent {
  overflow: hidden;  /* creates BFC → stops parent-child margin collapse */

/* BFC stops content wrapping around a float */
.float-sibling {
  overflow: hidden;  /* creates its own BFC → won't wrap around float */

CSS Performance

Advanced
Rendering Pipeline

Browser parses HTML → builds DOM. Parses CSS → builds CSSOM. Combined into Render Tree → Layout → Paint → Composite. Properties that trigger layout (top/left/width) are expensive; transform/opacity only trigger Composite.

GPU Composited Properties

Only transform and opacity are GPU-composited — they skip Layout and Paint. Animate these instead of top/left for smooth 60fps animations.

performance.css
/* ❌ Causes layout recalculation on every frame */
.bad-animation {
  transition: left 300ms;   /* triggers layout */
  transition: width 300ms;  /* triggers layout */
  transition: margin 300ms; /* triggers layout */
}

/* ✅ Compositor-only — fast, no layout/paint */
.good-animation {
  transition: transform 300ms ease;
  transition: opacity  300ms ease;
}

/* will-change — hints browser to promote to own layer */
.card:hover {
  will-change: transform;  /* pre-promote before hover */
}
/* ⚠ Don't use on everything — each layer uses VRAM */

/* contain — limit rendering scope */
.widget {
  contain: layout;    /* layout doesn't affect outside */
  contain: style;     /* counters don't leak out */
  contain: paint;     /* painting clipped to element */
  contain: strict;    /* layout + style + paint + size */
  contain: content;   /* layout + style + paint (common) */
}

/* content-visibility — skip rendering off-screen sections */
.section {
  content-visibility: auto;
  contain-intrinsic-size: 0 500px;  /* estimated height for scrollbar */
}

/* Selector performance */
/* ❌ Slow — universal descendant */
.nav * span { ... }
/* ✅ Fast — direct class */
.nav-link { ... }

⬡ Interview Q&A

Top CSS & SCSS Interview Questions

Must Prepare
What is the difference between display:none, visibility:hidden, and opacity:0?
display:none removes the element completely from the document flow — it takes no space, receives no events, and is invisible. visibility:hidden makes the element invisible but it still occupies its space in the layout (like an invisible box). opacity:0 makes the element fully transparent but it occupies space AND still receives pointer events (clicks, hovers). Use display:none to hide completely, opacity:0 when you want to animate a hide/show without layout shifts.
Explain the CSS Box Model and box-sizing: border-box.
Every element is a box made of content → padding → border → margin. By default (box-sizing: content-box), width refers only to the content area — so actual rendered width = width + padding + border. With box-sizing: border-box, width includes padding and border — the content area shrinks to compensate. This is much more intuitive for layout. Always apply *, *::before, *::after { box-sizing: border-box; } at the start of every project.
What is CSS specificity and how is it calculated?
Specificity determines which CSS rule wins when multiple rules target the same element. It's calculated as a three-part score (a, b, c): a = number of ID selectors, b = number of class/attribute/pseudo-class selectors, c = number of element/pseudo-element selectors. Higher a always beats lower a regardless of b and c. Inline styles override everything. !important overrides inline styles. So: #id (1,0,0) beats .class.class.class (0,3,0).
What is the difference between Flexbox and CSS Grid? When to use each?
Flexbox is one-dimensional — it works in a single direction (row OR column). It's content-first: you put items in and flexbox figures out the space. Great for navigation bars, toolbars, centering, button groups, and any single-direction distribution. CSS Grid is two-dimensional — rows AND columns simultaneously. It's layout-first: you define the grid structure, then place items. Perfect for full-page layouts, card grids, and anything requiring two-axis control. Often you use both: Grid for page layout, Flexbox for component internals.
What is a CSS stacking context and what creates one?
A stacking context is a three-dimensional rendering layer. Elements inside a stacking context are painted together, and their z-index values only compete within that context. A child with z-index:9999 inside a parent with z-index:1 will never appear above a sibling element that has z-index:2 — the entire parent (with all its children) is compared as one unit. Stacking contexts are created by: position + z-index (not auto), opacity < 1, transform, filter, will-change, isolation:isolate, and others.
What is margin collapsing and when does it happen?
Margin collapsing is when vertical margins between adjacent block elements merge into a single margin equal to the largest of the two (not their sum). It happens in three cases: (1) Adjacent siblings — bottom margin of one collapses with top margin of the next. (2) Parent-first child — if no border/padding/BFC separates them, parent's top margin collapses with child's top margin. (3) Empty blocks — if an element has no content, its top and bottom margins collapse. Horizontal margins never collapse. Flex/grid items don't collapse.
What is a Block Formatting Context (BFC) and what are its uses?
A BFC is an isolated layout environment where elements are laid out independently of the outside. It solves three classic problems: (1) Float containment — without BFC, floats collapse their parent's height; adding display: flow-root to the parent creates a BFC that contains floats. (2) Preventing margin collapse — elements inside a BFC don't collapse margins with outside elements. (3) Preventing float overlap — a BFC element won't wrap around adjacent floats. Creates BFC: overflow (not visible), display: flow-root/flex/grid, position: absolute/fixed, float.
Explain CSS variables vs SCSS variables. What are the key differences?
SCSS variables ($var) are compile-time — they're processed before the browser sees them and disappear after compilation. They can't be changed at runtime, can't inherit through the DOM, and can't be accessed with JavaScript. CSS custom properties (--var) are runtime — they live in the browser, cascade and inherit through the DOM, respond to media queries, can be changed with JavaScript (element.style.setProperty), and even support transitions. Best practice: use SCSS variables for values only needed at compile time (build configuration, calculations), and CSS variables for values that need to respond to context or be animated.
What is the difference between @mixin and @extend in SCSS?
@mixin copies the CSS into every place it's included — if used 10 times, the rules appear 10 times in the output. But mixins accept parameters, can be used anywhere, work inside media queries, and can contain @content blocks. @extend/%placeholder merges selectors — instead of copying, it adds your selector to the original rule. More DRY output but no parameters, can't be used inside media queries, and can create unexpected selector chains. General rule: prefer mixins for parameterized/reusable patterns; use placeholders for simple shared styles without parameters.
How would you implement a dark mode?
Best approach: define all colors as CSS custom properties on :root. Then either: (1) Use @media (prefers-color-scheme: dark) to override variables automatically based on OS setting. (2) Add a .dark class to <html> and override variables there — allows JS toggle. (3) Use both: auto-detect with media query, allow manual override with class, and persist the preference in localStorage. The key is never hardcode color values in component styles — always reference variables like var(--bg), so swapping the theme is just redefining the variables.
How do you center an element horizontally and vertically?
Method 1 — Flexbox (most common): display: flex; align-items: center; justify-content: center; on the parent. Method 2 — Grid: display: grid; place-items: center; on the parent. Method 3 — Absolute + Transform: position: absolute; top: 50%; left: 50%; transform: translate(-50%, -50%); — parent must be positioned. Method 4 — margin: auto (block elements, known width): width: fit-content; margin: 0 auto; — horizontal only. Method 5 — Grid with margin auto: display: grid; on parent + margin: auto; on child. Modern recommendation: use Flexbox or Grid place-items.
What are the differences between position: sticky and position: fixed?
position: fixed is always positioned relative to the viewport — it stays in the same place on screen regardless of scrolling. It's removed from the document flow (takes no space). position: sticky is a hybrid — it acts like relative while scrolling normally, but "sticks" to its offset (e.g., top: 0) when the user scrolls to it, then unsticks when its scroll container is scrolled past. It remains in the document flow (still takes space). Common sticky gotcha: it won't work if an ancestor has overflow: hidden/auto.
What CSS properties can be animated efficiently (GPU-accelerated)?
Only transform and opacity are compositor-layer properties — the browser can animate them entirely on the GPU, bypassing Layout and Paint entirely. This means they can animate at 60fps even if the main thread is busy. All other properties (top, left, width, height, background-color, margin) trigger layout recalculation on every frame, causing potential jank. Always animate transform: translateX() instead of left:, and opacity instead of visibility. Use will-change: transform to pre-promote an element before animation begins, but use it sparingly as each promoted layer consumes VRAM.
What is the 7-1 SCSS architecture pattern?
The 7-1 pattern organizes SCSS into 7 folders plus 1 main file that imports everything. The folders are: abstracts (variables, mixins, functions — no CSS output), base (reset, typography, global defaults), components (individual UI pieces like buttons, cards), layout (large-scale sections like header, footer, grid), pages (page-specific styles), themes (dark/light theme overrides), and vendors (third-party CSS). The main.scss imports from all folders using @use. This separation keeps files small, focused, and easy to find — especially valuable on large teams.
Explain the difference between :nth-child() and :nth-of-type().
:nth-child(n) counts all sibling elements regardless of type. p:nth-child(2) only matches a <p> that is the 2nd child of its parent — if the 2nd child is a <div>, nothing matches. :nth-of-type(n) counts only elements of the same type. p:nth-of-type(2) matches the 2nd <p> among all <p> tags in the parent, skipping non-paragraph siblings. Use nth-child when you know the DOM structure; use nth-of-type for more robust type-based selection.
What is the difference between @use and @import in SCSS? Why is @import deprecated?
@import (deprecated) loads files into a global namespace — every variable, mixin, and function from every imported file is available everywhere. This causes naming collisions and makes it impossible to know where a variable is defined. It also doesn't handle circular imports well. @use is the modern replacement — each file is loaded with a namespace (filename by default): variables.$primary. You explicitly see where things come from, there are no naming collisions, files are only loaded once even if @use'd multiple times, and private members (prefixed with - or _) aren't accessible from outside. Use @forward to re-export things from a folder's index file.
