📱 Core Concepts
01

What is React Native?

Foundation

React Native is a framework made by Facebook (Meta) that lets you build real mobile apps for iOS and Android using JavaScript and React. The key word is "real" — it does NOT make a website inside your app. It creates actual native mobile UI components.

NOT a WebView app

A WebView just shows a website inside an app shell (like Cordova/Ionic). React Native is completely different — it renders real native UI elements like a real Android/iOS app.

Write Once, Run on Both

One codebase for both iOS and Android. You write JavaScript, and React Native translates your components to native platform UI. Huge time and cost saving.

Uses React

Same concepts — components, state, props, hooks. If you know React for web, learning React Native is mostly about learning the mobile-specific APIs and components.

Access Native APIs

Camera, GPS, accelerometer, push notifications — all accessible. Not limited like a web app. You get the full power of the native platform.

Simple Analogy: Think of React Native as a translator. You write in JavaScript (English), and RN translates your instructions into native code (local language) that iOS and Android actually understand.

02

How React Native Works Internally

Interview Gold

This is a very common interview question. React Native has two worlds — the JavaScript world and the Native world. Understanding how they communicate is crucial.

Old Architecture (Bridge-based):

JS Thread
(your code runs here)
⟷
Bridge
(serialized JSON)
⟷
Native Thread
(iOS / Android)
JS Thread

Your React and JavaScript logic runs here. When you call setState or press a button, this thread processes the logic and sends instructions to the native side.

Native Thread

The actual Android/iOS thread that renders real UI components and handles native events (touch, gestures). This is what users see and interact with.

The Bridge

Connects JS ↔ Native. All messages are serialized (converted) to JSON and passed asynchronously. This JSON serialization is the main bottleneck — it's slow and can drop frames.

Shadow Thread

A third thread that calculates layout using Yoga (a cross-platform layout engine that implements Flexbox). Sits between JS and Native threads.

Bridge Bottleneck: The bridge is async and serializes everything to JSON. If you're doing heavy work or many rapid interactions, the bridge gets congested. This is why animations done purely in JS can stutter — the JS thread can't keep up at 60fps.
how-rn-works.js
// When you write this in JS:
<View style={{ backgroundColor: 'red' }} />

// React Native internally does something like:
// 1. JS creates a virtual node describing the View
// 2. Shadow thread calculates the layout (Yoga)
// 3. Bridge sends serialized message to native side
// 4. Native side creates android.view.View / UIView
// → User sees a real native component, not a web element

Styling & Flexbox

Core

React Native has no CSS. Styles are written as JavaScript objects using StyleSheet.create(). The names are similar to CSS but in camelCase. Flexbox is the default layout system — and in RN, the default flex direction is column (not row like web).

Key difference from web: In RN, flexDirection defaults to 'column' (top to bottom). On the web, it defaults to 'row'. Remember this in interviews!
styling.jsx
import { StyleSheet, View, Text } from 'react-native';

export default function App() {
  return (
    <View style={styles.container}>
      <View style={styles.box}>
        <Text style={styles.title}>Hello!</Text>
      </View>
    </View>
  );
}

const styles = StyleSheet.create({
  container: {
    flex: 1,                  // takes full screen height
    justifyContent: 'center', // vertical center (flexDirection: column)
    alignItems: 'center',    // horizontal center
    backgroundColor: '#fff',
  },
  box: {
    width: 200,
    height: 100,
    backgroundColor: '#6c63ff',
    borderRadius: 12,
    padding: 16,
    flexDirection: 'row',  // override to horizontal
    gap: 8,                 // RN 0.71+ supports gap
  },
  title: {
    fontSize: 18,
    fontWeight: '700',
    color: '#fff',
  }
});

05

State & Props

Core

Same as React web. Props = data passed from parent to child (read-only). State = data that lives inside a component and can change, causing re-renders. React Native uses the exact same hooks.

state-props.jsx
import { useState, useEffect } from 'react';
import { View, Text, TouchableOpacity } from 'react-native';

// Props: parent passes data down
function UserCard({ name, age }) { // destructured props
  return <Text>{name}, {age} years old</Text>;
}

// State: internal data that changes
function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    console.log('Count changed:', count);
    return () => console.log('Cleanup on unmount or re-run');
  }, [count]); // runs whenever count changes

  return (
    <View>
      <Text>Count: {count}</Text>
      <TouchableOpacity onPress={() => setCount(c => c + 1)}>
        <Text>Increment</Text>
      </TouchableOpacity>
    </View>
  );
}

Navigation (React Navigation)

Very Important

React Native has no built-in navigation. The most popular library is React Navigation. Think of it like a browser's back/forward but for mobile screens.

Stack Navigator

Screens stacked on top of each other. Go to a screen = push. Back = pop. Like a stack of cards. Used for: Login → Home → Detail.

Tab Navigator

Bottom tabs (or top) that switch between screens. Like Instagram's bottom tab bar. Each tab has its own stack.

Drawer Navigator

Slide-out side menu (hamburger menu). Common in admin or content-heavy apps.

navigation-setup.jsx
import { NavigationContainer } from '@react-navigation/native';
import { createNativeStackNavigator } from '@react-navigation/native-stack';
import { createBottomTabNavigator } from '@react-navigation/bottom-tabs';

const Stack = createNativeStackNavigator();
const Tab   = createBottomTabNavigator();

// Stack Navigation
function AppStack() {
  return (
    <Stack.Navigator initialRouteName="Home">
      <Stack.Screen name="Home"   component={HomeScreen} />
      <Stack.Screen name="Detail" component={DetailScreen} />
    </Stack.Navigator>
  );
}

// Navigating between screens
function HomeScreen({ navigation }) {
  return (
    <TouchableOpacity
      onPress={() => navigation.navigate('Detail', { id: 42, name: 'Alice' })}
    >
      <Text>Go to Detail</Text>
    </TouchableOpacity>
  );
}

// Receiving params in destination screen
function DetailScreen({ route }) {
  const { id, name } = route.params;
  return <Text>User: {name} (ID: {id})</Text>;
}

// navigation methods:
// navigate('Screen')   — go to screen (no duplicate)
// push('Screen')       — push even if duplicate
// goBack()             — go back
// replace('Screen')    — replace current screen
// reset({...})         — reset entire stack

Core

import { useState } from 'react';
import { TextInput, Button, Pressable, TouchableOpacity, Text, View } from 'react-native';

        value={password}
        onChangeText={setPassword}
        secureTextEntry               // hides password

        <Text>Touchable Button</Text>

        onPress={() => {}}
        style={({ pressed }) => ({ opacity: pressed ? 0.5 : 1 })}
      >
        <Text>Pressable Button</Text>
      </Pressable>
    </View>
  );
}

Lists — FlatList vs ScrollView

Very Important

This is asked in almost every React Native interview. Never use ScrollView for long lists. Use FlatList instead. Here's why:

ScrollView ❌ for long lists

Renders ALL children immediately — even items not visible on screen. If you have 1000 items, all 1000 are in memory. This causes memory issues and slow initial load.

FlatList ✅ for long lists

Only renders items currently visible on screen + a small buffer. As you scroll, old items are recycled and new ones are rendered. Like a window sliding over your data.

flatlist-example.jsx
import { FlatList, Text, View } from 'react-native';

const DATA = [
  { id: '1', name: 'Alice' },
  { id: '2', name: 'Bob' },
  // ... 1000 more items
];

function UserList() {
  return (
    <FlatList
      data={DATA}
      keyExtractor={item => item.id}    // unique key — required!
      renderItem={({ item }) => (        // render each row
        <View>
          <Text>{item.name}</Text>
        </View>
      )}
      initialNumToRender={10}           // how many to render first
      windowSize={5}                   // render 5x viewport height
      removeClippedSubviews             // unmount off-screen (Android)
      ListHeaderComponent={() => <Text>Users</Text>}  // header
      ListEmptyComponent={() => <Text>No users</Text>}  // empty state
      onEndReached={() => loadMore()}  // pagination trigger
      onEndReachedThreshold={0.5}       // trigger at 50% from bottom
    />
  );
}

API Calls

Core
api-calls.jsx
import { useState, useEffect } from 'react';
import { View, Text, ActivityIndicator } from 'react-native';

function UserScreen() {
  const [users, setUsers]     = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError]     = useState(null);

  useEffect(() => {
    const fetchUsers = async () => {
      try {
        setLoading(true);
        const res  = await fetch('https://api.example.com/users');
        if (!res.ok) throw new Error(`HTTP ${res.status}`);
        const data = await res.json();
        setUsers(data);
      } catch (e) {
        setError(e.message);
      } finally {
        setLoading(false);
      }
    };
    fetchUsers();
  }, []);

  if (loading) return <ActivityIndicator size="large" />;
  if (error)   return <Text>Error: {error}</Text>;

  return <Text>{users.length} users loaded</Text>;
}

Core

Sometimes iOS and Android need different code. React Native gives you tools to handle this cleanly without duplicating entire files.

platform-specific.jsx
import { Platform, StyleSheet } from 'react-native';

// Method 1: Inline Platform.OS
const isIOS     = Platform.OS === 'ios';
const isAndroid = Platform.OS === 'android';

const paddingTop = Platform.select({
  ios:     44,    // safe area on iPhone
  android: 24,    // status bar height on Android
  default: 0,
});

// Method 2: Platform-specific files
// Button.ios.js    → used automatically on iOS
// Button.android.js → used automatically on Android
// import Button from './Button'; ← RN picks the right file

// Method 3: In StyleSheet
const styles = StyleSheet.create({
  shadow: {
    ...Platform.select({
      ios:     { shadowColor: '#000', shadowOpacity: 0.3, shadowRadius: 4 },
      android: { elevation: 4 },  // Android uses elevation for shadows
    }),
  }

Performance Optimization

Very Important

The goal is to keep your app at 60fps (or 120fps on Pro devices) — that means each frame must render in under 16ms. Re-renders are the main enemy.

performance.jsx
import { memo, useCallback, useMemo } from 'react';

// 1. React.memo — skip re-render if props didn't change
const UserCard = memo(function UserCard({ name, age }) {
  return <Text>{name}, {age}</Text>;
});
// Now UserCard only re-renders when name or age changes

// 2. useCallback — memoize functions so they don't recreate each render
function Parent() {
  const [count, setCount] = useState(0);

  const handlePress = useCallback(() => {
    console.log('pressed');
  }, []); // ← empty deps = created once, never recreated

  // Without useCallback: every Parent re-render creates a NEW handlePress
  // → UserCard sees new prop → unnecessary re-render
  return <UserCard onPress={handlePress} />;
}

// 3. useMemo — memoize expensive computed values
const filteredList = useMemo(
  () => bigList.filter(item => item.active),
  [bigList] // only recomputes when bigList changes
);

// 4. Optimize FlatList
<FlatList
  data={data}
  renderItem={useCallback(({ item }) => <Row item={item} />, [])}
  keyExtractor={useCallback(item => item.id, [])}
  getItemLayout={(_, index) => ({  // skip layout calculation
    length: 60, offset: 60 * index, index
  })}
  maxToRenderPerBatch={10}
  windowSize={5}
/>
TechniqueWhat It DoesWhen to Use
React.memoSkip re-render if props unchangedPure components that render often
useCallbackMemoize function referencesFunctions passed as props to memo'd children
useMemoMemoize expensive computationsHeavy filtering, sorting, transformations
getItemLayoutPre-calculate item heights in FlatListFixed-height list rows
removeClippedSubviewsUnmount off-screen views (Android)Very long lists on Android
InteractionManagerDefer work until animations are doneHeavy work after navigation transition

Native Modules

Advanced

Sometimes React Native doesn't have a JS library for what you need. You can write code in Java/Kotlin (Android) or Swift/Objective-C (iOS) and expose it to JavaScript. This is called a Native Module.

Example use cases: Bluetooth LE communication, custom hardware SDKs, advanced camera controls, payment SDKs that only have native SDKs. Most common ones already have community packages (react-native-camera, react-native-ble-plx etc.).
native-module-usage.js
// After writing your native module in Java/Swift,
// you access it in JS like this:
import { NativeModules } from 'react-native';

const { MyCustomModule } = NativeModules;

// Call native method
MyCustomModule.doSomething('param')
  .then(result => console.log(result))
  .catch(err => console.error(err));

Async Storage (Local Storage for RN)

Core

React Native has no localStorage. Instead, use AsyncStorage — it's like localStorage but asynchronous (returns Promises). Good for storing small data like user preferences, tokens (use SecureStore for sensitive data).

async-storage.js
import AsyncStorage from '@react-native-async-storage/async-storage';

// Save data
await AsyncStorage.setItem('user', JSON.stringify({ name: 'Alice' }));

// Read data
const raw  = await AsyncStorage.getItem('user');
const user = raw ? JSON.parse(raw) : null;

// Delete
await AsyncStorage.removeItem('user');

// Clear everything
await AsyncStorage.clear();

// ⚠ For sensitive data (tokens, passwords) use:
// expo-secure-store  OR  react-native-keychain
// These use the device's secure enclave / Keystore

Animations

Advanced
Animated API (Built-in)

RN's built-in animation system. Works on JS thread by default. Good for basic fade, slide, scale. Use useNativeDriver: true to run on native thread for smooth 60fps.

React Native Reanimated

Third-party library that runs animations entirely on the native/UI thread. No bridge lag. The go-to for production apps. Required for complex gesture-based animations.

animations.jsx
import { Animated, useRef, useEffect } from 'react-native';

function FadeIn({ children }) {
  const opacity = useRef(new Animated.Value(0)).current;

  useEffect(() => {
    Animated.timing(opacity, {
      toValue: 1,
      duration: 500,
      useNativeDriver: true, // 🔑 MUST use for opacity/transform
    }).start();
  }, []);

  return (
    <Animated.View style={{ opacity }}>
      {children}
    </Animated.View>
  );
}

// Animated API types:
// Animated.timing()  — smooth over time
// Animated.spring()  — bouncy, physics-based
// Animated.decay()   — slows to a stop
// Animated.sequence() — run one after another
// Animated.parallel() — run simultaneously

Permissions Handling

Core
permissions.js
import { PermissionsAndroid, Platform } from 'react-native';
import { request, PERMISSIONS, RESULTS } from 'react-native-permissions';

async function requestCameraPermission() {
  const permission = Platform.OS === 'ios'
    ? PERMISSIONS.IOS.CAMERA
    : PERMISSIONS.ANDROID.CAMERA;

  const result = await request(permission);

  switch (result) {
    case RESULTS.GRANTED:
      console.log('Camera permission granted');
      break;
    case RESULTS.DENIED:
      console.log('Permission denied, can request again');
      break;
    case RESULTS.BLOCKED:
      console.log('Blocked — user must go to settings');
      break;
  }
}

// Common permissions: Camera, Microphone, Location,
// Contacts, Notifications, Photo Library, Bluetooth

Push Notifications (Firebase FCM)

Important

Push notifications are sent from your server → Firebase Cloud Messaging (FCM) → device. You need to handle: foreground (app open), background (app minimized), and killed (app closed) states differently.

push-notifications.js
import messaging from '@react-native-firebase/messaging';

// 1. Request permission (iOS requires explicit permission)
await messaging().requestPermission();

// 2. Get device token (send this to your backend)
const token = await messaging().getToken();
console.log('FCM Token:', token); // save to your server

// 3. Foreground — app is open
messaging().onMessage(async remoteMessage => {
  console.log('Foreground message:', remoteMessage.notification?.title);
  // Show in-app notification banner
});

// 4. Background / Killed — app is in background or closed
messaging().setBackgroundMessageHandler(async remoteMessage => {
  console.log('Background message received');
});

// 5. App opened FROM a notification tap
messaging().onNotificationOpenedApp(remoteMessage => {
  // Navigate to relevant screen
  navigation.navigate('Detail', { id: remoteMessage.data.id });
});

Deep Linking

Important

Deep linking lets you open a specific screen in your app from a URL — like clicking myapp://products/42 and landing directly on the product detail screen. Used in push notifications, emails, and web-to-app redirects.

deep-linking.jsx
// In your React Navigation config:
const linking = {
  prefixes: ['myapp://', 'https://myapp.com'],
  config: {
    screens: {
      Home:    'home',             // myapp://home
      Product: 'products/:id',    // myapp://products/42
      Profile: 'user/:username',  // myapp://user/alice
    }
  }
};

function App() {
  return (
    <NavigationContainer linking={linking}>
      // React Navigation handles the URL automatically
      // and navigates to the right screen
    </NavigationContainer>
  );
}

New Architecture 🔥 HOT

Must Know

This is the biggest change in React Native's history. The old Bridge was the main performance bottleneck. The new architecture replaces it with three key pieces: JSI, Fabric, and TurboModules.

Old vs New Architecture:

❌ Old Architecture (Bridge)

JS ↔ Bridge ↔ Native. All messages serialized to JSON. Async only. Copying data between threads is expensive. Bridge can get congested under heavy load. Limited synchronous access.

✅ New Architecture (JSI)

JS directly references native objects via C++ host objects. No serialization. No copying. Synchronous AND asynchronous calls possible. Much faster, no bridge bottleneck.

new-architecture-concepts.txt
┌─────────────────────────────────────────────────┐
│             NEW ARCHITECTURE STACK               │
├─────────────────────────────────────────────────┤
│  JavaScript (React + App Code)                  │
│         ↕ (JSI — direct C++ reference)          │
│  TurboModules  (native modules via JSI)          │
│  Fabric        (new renderer via JSI)            │
│         ↕                                        │
│  iOS Native / Android Native                    │
└─────────────────────────────────────────────────┘

JSI (JavaScript Interface):
  • C++ layer that lets JS hold direct references to native objects
  • No JSON serialization → sync calls are possible
  • JS can call native methods as if they were JS functions

Fabric:
  • New rendering system replacing the old UIManager
  • Renders UI on the native thread directly via JSI
  • Concurrent rendering (like React 18 Concurrent Mode)
  • Synchronous layout measurement possible

TurboModules:
  • New native modules system — lazy loaded (old: all loaded at startup)
  • Generated TypeScript types for type safety
  • Faster startup because only used modules are loaded

Codegen:
  • Auto-generates C++ glue code from TypeScript interfaces
  • Ensures type safety between JS and native at compile time
FeatureOld ArchitectureNew Architecture
CommunicationJSON over async bridgeC++ via JSI (sync + async)
SerializationAll data serialized to JSONDirect object references
Native ModulesAll loaded at startupTurboModules — lazy loaded
RendererUIManager (old)Fabric (new)
Sync callsNot possiblePossible via JSI
Startup timeSlower (all modules loaded)Faster (lazy loading)
React Native 0.76+ enables new architecture by default. You'll almost certainly be asked: "What is JSI and how does it differ from the Bridge?"

Hermes Engine 🔥

Important

Hermes is a JavaScript engine built by Meta specifically for React Native. It replaces JavaScriptCore (JSC — the engine used in Safari). Hermes is optimized for mobile — it's fast to start, uses less memory, and pre-compiles JS to bytecode at build time.

JavaScriptCore (old default)

General-purpose JS engine (used in Safari). Parses and compiles JS at runtime. Larger startup time. More memory usage.

Hermes (new default)

Pre-compiles JS to bytecode at BUILD time. At runtime, just executes bytecode — skips parsing and compilation. Much faster TTI (Time To Interactive) and lower memory.

AdvantageExplanation
Faster startup (TTI)Bytecode is pre-compiled — no JS parsing at runtime
Lower memory usageOptimized for resource-constrained mobile devices
Smaller APK / IPABytecode can be more compact than raw JS
Better performanceSpecifically tuned for RN workloads, not general browser JS
Hermes is enabled by default from RN 0.70+. Enable it in android/app/build.gradle with enableHermes: true.

FlatList Deep Dive

Interview Trap
flatlist-deep-dive.jsx
<FlatList
  data={data}
  renderItem={renderItem}
  keyExtractor={item => item.id}

  // — PERFORMANCE PROPS —

  initialNumToRender={10}
  // How many items to render for first paint.
  // Higher = slower initial load but less white flash on scroll

  windowSize={5}
  // Render window = 5x the viewport height.
  // Items outside this window are unmounted to save memory.
  // Default is 21. Lower = less memory, more blank flash.

  maxToRenderPerBatch={10}
  // How many items to render per JS batch update.
  // Higher = faster fill but could drop frames.

  updateCellsBatchingPeriod={50}
  // Milliseconds between batch render updates. Default 50ms.

  getItemLayout={(data, index) => ({
    length: 80,           // fixed row height
    offset: 80 * index,   // position of row
    index,
  })}
  // Skip layout calculation — HUGE perf boost for fixed-height rows.
  // Also enables scrollToIndex without measuring all items first.

  removeClippedSubviews
  // Unmount views off-screen on Android. Memory saver for very long lists.

  keyExtractor={useCallback(item => item.id, [])}
  renderItem={useCallback(({ item }) => <Row item={item} />, [])}
  // Wrap both in useCallback to prevent recreating on every parent render
/>
Common FlatList mistake: Putting renderItem inline as an arrow function (not wrapped in useCallback). Every parent re-render creates a new function reference → FlatList thinks renderItem changed → re-renders all visible rows unnecessarily.

Interview Trap
memory-leaks.jsx
// ⚠ MEMORY LEAK — setting state after component unmounts
useEffect(() => {
  fetch('/api/data')
    .then(res => res.json())
    .then(data => setData(data)); // component may have unmounted!
}, []);

// ✅ FIX 1: AbortController to cancel fetch on unmount
useEffect(() => {
  const controller = new AbortController();

  fetch('/api/data', { signal: controller.signal })
    .then(res => res.json())
    .then(data => setData(data))
    .catch(e => { if (e.name !== 'AbortError') throw e; });

  return () => controller.abort(); // cleanup on unmount
}, []);

// ✅ FIX 2: Mounted flag
useEffect(() => {
  let mounted = true;
  fetch('/api').then(res => res.json()).then(data => {
    if (mounted) setData(data); // safe check
  });
  return () => { mounted = false; };
}, []);

// ✅ Always clean up subscriptions, timers, event listeners
useEffect(() => {
  const sub = NetInfo.addEventListener(handler);
  const timer = setInterval(tick, 1000);

    sub();              // unsubscribe

Error Boundaries

Interview Trap

Error Boundaries catch JavaScript errors anywhere in the component tree and display a fallback UI instead of crashing the whole app. They must be class components (no hook equivalent yet).

error-boundary.jsx
import React from 'react';
import { Text, View } from 'react-native';

class ErrorBoundary extends React.Component {
  state = { hasError: false, error: null };

  static getDerivedStateFromError(error) {
    return { hasError: true, error };
  }

  componentDidCatch(error, info) {
    // Log to crash reporting (Sentry, Firebase Crashlytics)
    console.error('Caught error:', error, info);
  }

  render() {
    if (this.state.hasError) {
      return (
        <View>
          <Text>Something went wrong.</Text>
          <Text>{this.state.error?.message}</Text>
        </View>
      );
    }
    return this.props.children;
  }
}

// Wrap your app or specific sections
function App() {
  return (
    <ErrorBoundary>
      <MainApp />
    </ErrorBoundary>
  );
}

Security

Important
❌ Never use AsyncStorage for secrets

AsyncStorage is NOT encrypted. Anyone with physical device access or on a rooted device can read it. Never store JWT tokens, passwords, or API keys here.

✅ Use Secure Storage for secrets

react-native-keychain or expo-secure-store use iOS Keychain / Android Keystore — hardware-backed encrypted storage. Always use for tokens and credentials.

security.js
import * as Keychain from 'react-native-keychain';

// Save credentials securely
await Keychain.setGenericPassword('username', 'jwtTokenHere');

// Read credentials
const creds = await Keychain.getGenericPassword();
if (creds) {
  const token = creds.password;
}

// Delete
await Keychain.resetGenericPassword();

      renderItem={renderItem}

  );

Top Interview Q&A

Must Prepare
Q: How does React Native work internally?
React Native runs JS on a separate thread. When your React code renders components, the Shadow Thread calculates layout using Yoga (Flexbox). These layout instructions are sent to the Native Thread via the Bridge (old arch) or directly via JSI (new arch). The Native Thread then creates real native UI components — UIView on iOS, android.view.View on Android. All communication between JS and Native is asynchronous (old) or can be synchronous (new arch/JSI).
Q: What is the Bridge and why is it a bottleneck?
The Bridge is a message bus between the JS and Native threads. All messages are serialized to JSON strings, sent asynchronously, then deserialized on the other side. This serialization is expensive — it copies all data and can't handle synchronous calls. Under heavy load (rapid gestures, animations), the bridge gets congested. This is why animations done purely in JS can drop frames at 60fps.
Q: Why is FlatList better than ScrollView for long lists?
ScrollView renders ALL children at once, regardless of whether they're visible. For 1000 items, all 1000 are in memory. FlatList uses virtualization — it only renders items in or near the visible viewport. Off-screen items are unmounted and their memory freed. As you scroll, it recycles item views (like Android's RecyclerView). This makes FlatList memory-efficient and fast even for thousands of items.
Q: What is JSI and how does it differ from the Bridge?
JSI (JavaScript Interface) is a C++ layer that lets the JS engine directly reference native objects without serializing to JSON. Instead of sending messages through the bridge, JS holds a direct C++ reference (host object) to native objects. This means: no JSON serialization, synchronous calls are possible, much lower latency. The bridge required copying all data; JSI allows shared memory references.
Q: What is Fabric?
Fabric is the new rendering system in React Native's new architecture. It replaces the old UIManager. Fabric uses JSI to communicate directly with the native renderer, enabling synchronous layout measurements, better concurrency (works with React 18's concurrent features), and allows rendering to happen on the main thread with direct access to the shadow tree without bridge overhead.
Q: What are the advantages of Hermes?
Hermes is Meta's JS engine designed specifically for React Native. It pre-compiles JavaScript to bytecode at build time (not runtime). Benefits: faster Time To Interactive (TTI) because no runtime JS parsing/compilation, lower memory consumption (optimized for mobile), smaller bundle sizes, and generally better performance for RN workloads compared to JavaScriptCore.
Q: How do you handle offline mode?
Use NetInfo to detect connectivity changes. Cache API responses in AsyncStorage with timestamps. On app load, show cached data immediately (stale-while-revalidate pattern), then fetch fresh data. Queue mutations (POST/PUT/DELETE) when offline and execute them when connectivity is restored. Show a visible offline indicator so users know they're offline. Libraries like React Query or SWR handle much of this automatically.
Q: How do you secure tokens in React Native?
Never use AsyncStorage for tokens — it's unencrypted. Use react-native-keychain (uses iOS Keychain / Android Keystore — hardware-backed encrypted storage). For extra security: implement token rotation, use short-lived access tokens + refresh tokens, implement SSL pinning to prevent MITM attacks, detect rooted/jailbroken devices, and never hardcode secrets in source code.
Q: How do you pass data between screens?
Three main ways: (1) Navigation params — pass via navigate('Screen', { id: 42 }), receive via route.params. Good for lightweight data. (2) Global state — Redux, Zustand, Context API. Both screens read from the same store. Best for data used across many screens. (3) Navigation state / parent state. Avoid passing large objects via params — pass only IDs, let the destination screen fetch its own data.
Q: How do you prevent memory leaks in React Native?
Always return cleanup functions from useEffect: unsubscribe from event listeners (NetInfo, AppState), clear timers (clearTimeout, clearInterval), cancel in-flight API requests with AbortController, and use mounted flags to prevent setState after unmount. For FlatList, use removeClippedSubviews and proper windowSize. For images, use lazy loading and avoid storing large bitmaps in state.
