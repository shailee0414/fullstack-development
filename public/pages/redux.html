R-01

What is Redux & Why Use It

Core

What is Redux?

Redux is a predictable state container for JavaScript apps. It stores the entire application state in a single JavaScript object (the store), and state can only be changed by dispatching actions through pure reducer functions.

  • Created by Dan Abramov & Andrew Clark (2015), inspired by Flux + Elm
  • Works with React, Angular, Vue, or plain JS
  • State is read-only — you never mutate it directly
  • Changes are made by pure functions (reducers) — same input = same output, always
Interview Gold "Redux solves three problems: (1) State that needs to be shared across many components, (2) Props drilling through many layers, (3) Debugging — you can time-travel through every state change because all changes are logged via actions."

Unidirectional Data Flow

UI Event
→
dispatch(action)
→
Middleware
→
Reducer
→
New State
→
UI re-renders

Data flows in ONE direction — this makes state changes predictable and easy to trace. Compare to two-way binding (Angular) where state can change from multiple directions.

Q: Redux vs Context API — which to use?

Context: built-in, no setup, good for infrequent updates (theme, auth, locale). Every consumer re-renders on any context value change. Redux: dedicated devtools, time-travel debugging, middleware pipeline, selective subscriptions (useSelector only re-renders when selected slice changes), predictable with large teams. Use Context for simple global data, Redux for complex, frequently-updated state in large apps.

Redux Architecture

Core

The Three Pillars: Store, Action, Reducer

Store — Single Source of Truth
import { createStore } from 'redux'; // classic (legacy)

const store = createStore(rootReducer);

// The store has 4 methods:
store.getState();           // read current state
store.dispatch(action);    // trigger state change
store.subscribe(listener); // listen for changes
store.replaceReducer(fn);  // hot reload, code splitting
Action — What Happened
// Action = plain object describing an event
// MUST have a type property
const addUserAction = {
  type: 'user/addUser',    // required — describes what happened
  payload: {              // optional — carries data
    id: 1,
    name: 'Shailesh',
    role: 'Tech Lead'
  }
};

// Action types as constants (prevents typos)
const ADD_USER    = 'user/ADD_USER';
const DELETE_USER = 'user/DELETE_USER';
const UPDATE_USER = 'user/UPDATE_USER';
Reducer — How State Changes
// Reducer = pure function: (state, action) => newState
const initialState = { users: [], loading: false, error: null };

function userReducer(state = initialState, action) {
  switch (action.type) {
    case 'user/ADD_USER':
      return {
        ...state,
        users: [...state.users, action.payload]  // ✅ immutable
      };

    case 'user/DELETE_USER':
      return {
        ...state,
        users: state.users.filter(u => u.id !== action.payload)
      };

    case 'user/UPDATE_USER':
      return {
        ...state,
        users: state.users.map(u =>
          u.id === action.payload.id ? { ...u, ...action.payload } : u
        )
      };

    default:
      return state; // ✅ always return state for unknown actions
  }
}

Dispatch — Trigger Changes

Dispatch Flow
// dispatch sends action through middleware → reducer → new state
store.dispatch({ type: 'user/ADD_USER', payload: { id: 1, name: 'Shailesh' } });

// In React components (with react-redux)
const dispatch = useDispatch();
dispatch({ type: 'user/ADD_USER', payload: newUser });

Data Operations — Immutable Patterns

Core

All Immutable Update Patterns

Add, Update, Delete — Without Mutation
// ── OBJECTS ──────────────────────────────────────

// Add/update a field
return { ...state, name: 'New Name', loading: false };

// Remove a field
const { unwantedField, ...rest } = state;
return rest;

// Update nested object
return {
  ...state,
  user: { ...state.user, role: 'admin' }  // spread nested
};

// ── ARRAYS ───────────────────────────────────────

// ADD item
return { ...state, items: [...state.items, newItem] };

// DELETE item by id
return { ...state, items: state.items.filter(i => i.id !== id) };

// UPDATE item
return {
  ...state,
  items: state.items.map(i => i.id === id ? { ...i, ...changes } : i)
};

// INSERT at specific index
return {
  ...state,
  items: [...state.items.slice(0, idx), newItem, ...state.items.slice(idx)]
};

// ❌ NEVER DO THIS — direct mutation
state.users.push(newUser);       // WRONG
state.user.name = 'New';          // WRONG
state.users[0] = updatedUser;    // WRONG
Why Immutability Matters in Redux Redux checks if state changed using reference equality (===). If you mutate state directly, the reference stays the same → Redux thinks nothing changed → React doesn't re-render. Always return a NEW object/array.

R-04

Reducer Concepts

Core

Pure Functions — The Rule

A pure function: (1) Same input → always same output. (2) No side effects — no API calls, no console.log that changes state, no Date.now(), no Math.random().

Pure vs Impure Reducers
// ✅ PURE — deterministic, no side effects
function counterReducer(state = 0, action) {
  switch (action.type) {
    case 'INCREMENT': return state + 1;
    default: return state;
  }
}

// ❌ IMPURE — side effects in reducer
function badReducer(state = {}, action) {
  if (action.type === 'SAVE') {
    fetch('/api/save', ...);  // ❌ API call in reducer
    localStorage.setItem(...);   // ❌ side effect
    return { ...state, savedAt: new Date() }; // ❌ non-deterministic
  }
}

// ❌ IMPURE — mutates state
function mutatingReducer(state = [], action) {
  if (action.type === 'ADD') {
    state.push(action.payload); // ❌ mutates state
    return state;               // ❌ same reference
  }
}
Q: Why must reducers be pure functions?

Three reasons: (1) Predictability — same action always produces same state change, making bugs reproducible. (2) Time-travel debugging — DevTools can replay actions; impure functions would give different results on replay. (3) Testing — pure functions are trivially testable: call reducer(state, action), assert result. No mocking needed.

Integration with React

React

useSelector & useDispatch

Full React + Redux Integration
// 1. Setup Provider (index.js / App.js)
import { Provider } from 'react-redux';
import { store } from './store';

ReactDOM.render(
  <Provider store={store}>
    <App />
  </Provider>,
  document.getElementById('root')
);

// 2. useSelector — read from store
function UserList() {
  // Only re-renders when users array changes (not other state)
  const users = useSelector(state => state.user.users);
  const loading = useSelector(state => state.user.loading);

  // ❌ Avoid selecting entire state — over-selects
  const entire = useSelector(state => state); // re-renders on ANY change

  if (loading) return <p>Loading...</p>;
  return <ul>{users.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
}

// 3. useDispatch — send actions to store
function AddUserForm() {
  const dispatch = useDispatch();
  const [name, setName] = useState('');

  const handleAdd = () => {
    dispatch({
      type: 'user/ADD_USER',
      payload: { id: Date.now(), name }
    });
    setName('');
  };

  return (
    <div>
      <input value={name} onChange={e => setName(e.target.value)} />
      <button onClick={handleAdd}>Add User</button>
    </div>
  );
}
useSelector Re-render Behaviour useSelector runs on every store update. It compares the previous and new return value with ===. If different → component re-renders. For objects: { ...obj } !== { ...obj } even if contents are same. Use reselect or shallowEqual to avoid unnecessary re-renders.

createSlice — The Core of RTK

createSlice — Full Example
import { createSlice, createAsyncThunk } from '@reduxjs/toolkit';

// Async thunk — handles API call lifecycle automatically
export const fetchUsers = createAsyncThunk(
  'users/fetchAll',         // action type prefix
  async (_, { rejectWithValue }) => {
    try {
      const res = await fetch('/api/users');
      return await res.json();
    } catch (err) {
      return rejectWithValue(err.message); // ✅ structured error
    }
  }
);

const userSlice = createSlice({
  name: 'users',
  initialState: {
    list: [],
    loading: false,
    error: null,
    selectedUser: null
  },

  // Synchronous reducers — Immer handles immutability!
  reducers: {
    addUser: (state, action) => {
      state.list.push(action.payload);  // ✅ "mutation" OK — Immer converts to immutable
    },
    removeUser: (state, action) => {
      state.list = state.list.filter(u => u.id !== action.payload);
    },
    updateUser: (state, action) => {
      const idx = state.list.findIndex(u => u.id === action.payload.id);
      if (idx !== -1) state.list[idx] = action.payload;
    },
    resetUsers: () => ({ list: [], loading: false, error: null, selectedUser: null })
  },

  // Handle async thunk lifecycle
  extraReducers: (builder) => {
    builder
      .addCase(fetchUsers.pending, (state) => {
        state.loading = true;
        state.error = null;
      })
      .addCase(fetchUsers.fulfilled, (state, action) => {
        state.loading = false;
        state.list = action.payload;
      })
      .addCase(fetchUsers.rejected, (state, action) => {
        state.loading = false;
        state.error = action.payload;
      });
  }
});

// Auto-generated action creators
export const { addUser, removeUser, updateUser, resetUsers } = userSlice.actions;
export default userSlice.reducer;
How Immer Works in RTK createSlice wraps reducers with Immer. Immer creates a Proxy over your state. You "mutate" the proxy — Immer tracks all changes and produces a new immutable state object. You write simpler code; immutability is guaranteed under the hood.

Middleware — Internals

🔥 Most Asked

What is Middleware?

Middleware intercepts every action dispatched before it reaches the reducer. It's a pipeline where each piece of middleware can: inspect the action, transform it, stop it, delay it, or dispatch new actions.

dispatch(action)
→
Middleware 1
(logger)
→
Middleware 2
(thunk)
→
Middleware 3
(saga)
→
Reducer

Middleware Signature — The Curried Pattern

Custom Middleware from Scratch
// Middleware signature: store => next => action => result
// Three curried functions — Redux calls them in sequence

// 1. Logger Middleware
const loggerMiddleware = (store) => (next) => (action) => {
  console.group(action.type);
  console.log('Prev state:', store.getState());
  console.log('Action:', action);

  const result = next(action); // ← calls next middleware (or reducer)

  console.log('Next state:', store.getState());
  console.groupEnd();
  return result;
};

// 2. Timer Middleware — measure slow actions
const timerMiddleware = (store) => (next) => (action) => {
  const start = performance.now();
  const result = next(action);
  const duration = performance.now() - start;
  if (duration > 16) console.warn(`Slow action: ${action.type} (${duration}ms)`);
  return result;
};

// 3. Apply middleware (classic Redux)
import { createStore, applyMiddleware } from 'redux';
const store = createStore(
  rootReducer,
  applyMiddleware(loggerMiddleware, timerMiddleware)
);
Q: How does middleware work internally in Redux?

applyMiddleware uses function composition. Each middleware wraps the next with its own next function. store.dispatch is replaced with the composed chain. When you call dispatch(action), it enters the first middleware's (store)(next)(action). 'next' is the next middleware's dispatch. The last 'next' calls the actual reducer. This is the Chain of Responsibility pattern.

Redux Thunk & Thunk vs Saga

🔥 Most Asked

Redux Thunk — Dispatch Functions

Thunk middleware allows you to dispatch a function instead of an action object. That function receives dispatch and getState, so it can do async work then dispatch real actions.

Thunk — Async Action Creator
// Without thunk: can only dispatch objects
dispatch({ type: 'FETCH_USER' }); // sync only

// With thunk: dispatch a function
const fetchUser = (userId) => async (dispatch, getState) => {
  dispatch({ type: 'user/loading' });

  try {
    const res = await fetch(`/api/users/${userId}`);
    const data = await res.json();
    dispatch({ type: 'user/fetchSuccess', payload: data });
  } catch (err) {
    dispatch({ type: 'user/fetchFail', payload: err.message });
  }
};

// Component just dispatches it like any action
dispatch(fetchUser('user123'));

// Thunk middleware implementation (simplified)
const thunk = (store) => (next) => (action) => {
  if (typeof action === 'function') {
    // It's a thunk — call it with dispatch and getState
    return action(store.dispatch, store.getState);
  }
  return next(action); // plain object → pass through
};

Thunk vs Saga — Full Comparison

AspectRedux ThunkRedux Saga
ConceptDispatch functionsSeparate process listening for actions
Async syntaxasync/awaitGenerator functions (yield)
Bundle size~800 bytes 🏆~25KB
Learning curveLow — just async/awaitHigh — generators, effects
Complex flowsHard — nested callbacksEasy — sequential yields
CancellationManual (AbortController)Built-in (cancel, race, takeLatest)
TestingNeeds API mockingPure effect testing — no mocks needed
Side effectsInside thunk functionsDeclarative effects (call, put, take)
Best forSimple async ops, small appsComplex flows, real-time, background tasks
Q: When would you pick Thunk over Saga and vice versa?

Thunk: small-to-medium app, team unfamiliar with generators, simple CRUD async operations, bundle size matters. Saga: complex async flows (multi-step wizards, polling, WebSocket handling), need cancellation (user navigates away mid-request), real-time features, need debouncing built-in, or need parallel/sequential coordination of multiple async ops. At PB Fintech scale — Saga for critical flows (policy purchase), Thunk for simple lookups.

combineReducers

Core

Splitting State into Slices

combineReducers
import { combineReducers } from 'redux';
// OR in RTK: configureStore handles this automatically

// Each reducer manages its own slice of state
const rootReducer = combineReducers({
  auth:   authReducer,    // state.auth
  users:  userReducer,    // state.users
  orders: orderReducer,   // state.orders
  ui:     uiReducer,      // state.ui
});

// Resulting state shape:
{
  auth:   { isLoggedIn: true, user: {...} },
  users:  { list: [...], loading: false },
  orders: { items: [...], total: 0 },
  ui:     { theme: 'dark', sidebarOpen: false }
}

// How combineReducers works internally:
function combineReducers(reducers) {
  return function(state = {}, action) {
    return Object.keys(reducers).reduce((newState, key) => {
      newState[key] = reducers[key](state[key], action);
      // Every action is dispatched to EVERY reducer
      return newState;
    }, {});
  };
}
// Key insight: ALL reducers receive ALL actions — reducers just ignore irrelevant ones via default
Important: Every action hits every reducer This is why the default: return state case is mandatory. If you forget it, your reducer returns undefined on unrelated actions, wiping that slice of state.

Reselect & Memoized Selectors

Performance

createSelector — Memoized Derived State

reselect — Avoid Unnecessary Re-renders
import { createSelector } from 'reselect'; // or '@reduxjs/toolkit'

// Basic selectors (input selectors)
const selectUsers = state => state.users.list;
const selectFilter = state => state.ui.searchTerm;

// ❌ Without reselect — new array EVERY render → useSelector always triggers re-render
const selectActive = state => state.users.list.filter(u => u.active);

// ✅ With createSelector — memoized, only recomputes if inputs change
const selectActiveUsers = createSelector(
  [selectUsers, selectFilter],
  (users, filter) => {
    console.log('recomputing...'); // only logs when users or filter changes
    return users
      .filter(u => u.active)
      .filter(u => u.name.toLowerCase().includes(filter.toLowerCase()));
  }
);

// In component — only re-renders when selectActiveUsers output changes
const activeUsers = useSelector(selectActiveUsers);

// Parameterized selector (factory pattern)
const makeSelectUserById = (userId) => createSelector(
  [selectUsers],
  (users) => users.find(u => u.id === userId)
);
// Each component gets its own memoized selector instance
Q: Why is useSelector(state => state.users.filter(...)) problematic?

filter() always returns a new array, even if the elements are identical. useSelector compares results with ===. New array !== new array → component re-renders on every store update, even unrelated ones. createSelector caches the result and only recomputes if the input selectors' values change. RTK includes createSelector from reselect built-in.

Generator Functions — The Foundation

🔥 Most Asked

Why Generators? How They Work

Generator functions (function*) can pause and resume execution. yield pauses the function and returns a value. The caller calls .next() to resume. This makes async code look sequential without async/await.

Generator Function Internals
// Generator function returns an iterator
function* counter() {
  console.log('start');
  yield 1;           // pause, return 1
  console.log('resumed');
  yield 2;           // pause, return 2
  return 'done';     // final return
}

const gen = counter();        // doesn't run yet!
gen.next(); // logs 'start', returns { value: 1, done: false }
gen.next(); // logs 'resumed', returns { value: 2, done: false }
gen.next(); // returns { value: 'done', done: true }

// You can PASS values back in via next(value)
function* dialog() {
  const name = yield 'What is your name?';
  const age  = yield `Hello ${name}! How old are you?`;
  return `${name} is ${age} years old.`;
}
const d = dialog();
d.next();               // → 'What is your name?'
d.next('Shailesh');     // name = 'Shailesh' → 'Hello Shailesh! How old are you?'
d.next(28);             // age = 28 → 'Shailesh is 28 years old.'
Why Saga uses generators (not async/await) Saga's middleware takes control of the generator. When you yield an effect like call(api.fetchUser), the middleware runs the API call and passes the result back via .next(result). This lets saga intercept and control every async operation — enabling cancellation, testing, and retry logic that's impossible with async/await.
Q: Why does Redux Saga use generators instead of async/await?

Three reasons: (1) Testability — you can test sagas by manually calling .next() and asserting the yielded effects WITHOUT running actual API calls. (2) Cancellability — sagas can be cancelled mid-execution; async/await cannot be cancelled. (3) Declarative effects — yield call(fn) tells saga "please call this" rather than calling it directly, letting middleware control execution.

fork vs call — Blocking vs Non-Blocking

fork vs call — Practical
import { call, fork, spawn, join } from 'redux-saga/effects';

// call — BLOCKING (sequential)
function* sequential() {
  const user = yield call(fetchUser);     // waits for user
  const posts = yield call(fetchPosts, user.id); // then fetches posts
}

// fork — NON-BLOCKING (parallel background tasks)
function* withBackground() {
  const task = yield fork(syncDataInBackground); // starts, doesn't wait
  yield call(doMainWork);                          // continues immediately
  yield join(task);                                // optionally wait for fork
}

// spawn — independent, errors/cancellation don't propagate to parent
function* rootSaga() {
  yield spawn(analyticsTracker); // if this crashes, root saga continues
  yield spawn(watchUserActions);
}
Q: What is the difference between call and fork?

call is blocking — the saga pauses at yield call(fn) until fn resolves, like await. fork is non-blocking — the saga continues to the next line immediately and the forked task runs in background. Error in forked task propagates to parent (unlike spawn). Use call for sequential dependent operations, fork for background tasks or parallel execution.

all() — Parallel API Calls

all() — Like Promise.all

const profile = yield call(fetchProfile);   // 300ms
// Total: 900ms

// ✅ Parallel — fast (all run simultaneously)

  call(fetchUser),
  call(fetchOrders),
  call(fetchProfile)
]);
// Total: ~300ms (slowest one)

// all with object syntax

  user:   call(fetchUser),
  orders: call(fetchOrders)

Error Handling in Sagas

Critical

try/catch + Global Error Handling

Error Handling Patterns
// ── Pattern 1: try/catch in worker (most common) ──
function* fetchPolicyWorker({ payload: policyId }) {
  try {
    const policy = yield call(policyAPI.get, policyId);
    yield put({ type: 'policy/success', payload: policy });
  } catch (error) {
    yield put({ type: 'policy/error', payload: error.message });
    // Optional: show toast notification
    yield put({ type: 'ui/showToast', payload: { type: 'error', msg: error.message } });
  }
}

// ── Pattern 2: Global error handler ───────────────
function* globalErrorHandler() {
  while (true) {
    const { payload: error } = yield take('*ERROR*'); // watch all error actions
    yield call(errorReportingService.log, error);
    yield put({ type: 'ui/showGlobalError', payload: error });
  }
}

// ── Pattern 3: Retry logic ────────────────────────
function* retryWorker(action, retries = 3) {
  for (let i = 0; i < retries; i++) {
    try {
      const result = yield call(api.fetchData, action.payload);
      yield put(success(result));
      return;
    } catch (err) {
      if (i === retries - 1) yield put(failed(err.message));
      else yield delay(1000 * (i + 1)); // exponential backoff
    }
  }
}

S-07

Cancellation, take, and race

🔥 Senior Level

Cancelling Tasks + race()

cancel + race — Practical Patterns
import { fork, cancel, take, race, call, put, delay } from 'redux-saga/effects';

// ── Manual task cancellation ───────────────────
function* bgSyncSaga() {
  while (true) {
    yield take('SYNC_START');
    const task = yield fork(syncDataTask);

    yield take('SYNC_STOP');     // wait for stop action
    yield cancel(task);           // cancel the fork
  }
}

// ── race() — whoever finishes FIRST wins ──────
// Others are automatically cancelled!
function* fetchWithTimeout(action) {
  const { data, timeout } = yield race({
    data:    call(api.fetchUser, action.payload),
    timeout: delay(5000)          // 5 second timeout
  });

  if (timeout) {
    yield put(fetchTimedOut());
  } else {
    yield put(fetchSuccess(data));
  }
}

// ── race() for authentication flow ────────────
function* authFlow() {
  while (true) {
    yield take('LOGIN_REQUEST');

    const { success, logout } = yield race({
      success: call(loginFlow),
      logout:  take('LOGOUT')     // if user logs out mid-login → cancel
    });

    if (success) yield put(loginSuccess());
    if (logout)  yield put(logoutSuccess());
  }
}
Q: How do you handle race conditions in Sagas?

takeLatest automatically handles the most common race condition — it cancels the previous saga if a new action comes in before it finishes. For custom race conditions, use race() — yield race({ result: call(api), timeout: delay(5000) }) — the loser is automatically cancelled. For complex flows, fork + cancel gives full manual control.

Debounce / Throttle Patterns
import { takeLatest, call, put, delay, throttle } from 'redux-saga/effects';

// ── Debounce — wait until user stops typing ────
function* searchWorker({ payload: query }) {
  yield delay(400);             // wait 400ms before API call
  const results = yield call(searchAPI, query);
  yield put(searchSuccess(results));
}

// takeLatest cancels previous saga when new action arrives
// + delay inside = debounce effect!
function* watchSearch() {
  yield takeLatest('search/QUERY_CHANGE', searchWorker);
  // Rapid keystrokes → previous workers cancelled → delay resets → 1 API call
}

// ── Throttle — built-in effect ─────────────────
function* watchScroll() {
  // Max 1 call per 300ms, ignores rest
  yield throttle(300, 'SCROLL_EVENT', handleScrollWorker);
}

// ── takeLeading — first wins, ignore until done ─
import { takeLeading } from 'redux-saga/effects';
function* watchSubmit() {
  // Prevent double-submit: ignores new actions until current completes
  yield takeLeading('form/SUBMIT', submitFormWorker);
}
takeEvery vs takeLatest vs takeLeading vs throttle takeEvery: all parallel (logs, analytics). takeLatest: cancel prev, run new (search). takeLeading: ignore new until current done (form submit). throttle: rate limit (scroll events). These 4 cover 95% of real-world use cases.

Redux Questions

Q: Difference between Redux and Context API?

Context: built-in React, no setup, re-renders ALL consumers when value changes, no devtools, no middleware. Redux: separate library, boilerplate, but: selective re-renders (useSelector with equality check), time-travel DevTools, middleware pipeline (logging, analytics, async), predictable state with actions, designed for complex state. Use Context for simple global values (theme, auth). Use Redux for complex, frequently-updating shared state in large apps.

Q: Why must reducers be pure functions?

Predictability: same action always produces same state — bugs are reproducible. Time-travel debugging: DevTools replays all actions; impure functions give different results on replay. Testability: just call reducer(state, action) and assert — no mocks needed. Performance: Redux uses reference equality (===) to detect changes; if you mutate state, the reference is the same and React won't re-render.

Q: What is immutability and why does Redux require it?

Immutability means never modifying existing data — always creating new copies. Redux (and React) detect changes via reference equality (===). If you mutate state directly (state.push(item)), the reference stays the same → Redux thinks nothing changed → no re-render. Always return new objects/arrays: { ...state, field: newValue } or [...arr, newItem]. RTK uses Immer internally so you can write mutation-style code safely.

Q: Redux Toolkit vs Traditional Redux — what changed?

Traditional Redux requires: separate action type constants, separate action creator functions, switch/case reducer with spread operators everywhere, manual store setup with applyMiddleware. RTK simplifies: createSlice combines all three into one, Immer handles immutability so you can write direct mutations, configureStore sets up middleware + DevTools automatically, createAsyncThunk handles async lifecycle. RTK is the official recommendation — no reason to use traditional Redux in new code.

Q: Redux Thunk vs Redux Saga — detailed comparison?

Thunk: dispatch a function, use async/await inside, small bundle (~800 bytes), easy to learn, limited to the thunk function scope. Good for simple async. Saga: separate background process using generators, declarative effects (call, put, take), built-in cancellation (takeLatest, cancel, race), handles complex flows elegantly, excellent testing without mocks, ~25KB. Good for: polling, WebSockets, complex multi-step flows, cancellation. Choose Thunk for simplicity; Saga for power.

Q: How does Redux work internally?

Store holds state as a plain JS object. dispatch(action) passes action through middleware chain (each middleware calls next). After all middleware, action reaches the root reducer (via combineReducers, each slice reducer gets its own state slice and the action). Reducer returns new state. Store replaces old state. All subscribers (useSelector) are notified. useSelector runs selector, compares result with previous via ===, re-renders if different.

Q: What is createAsyncThunk and how does it handle lifecycle?

createAsyncThunk(actionTypePrefix, payloadCreator) generates three action types automatically: pending (starts), fulfilled (success), rejected (failure). When dispatched, it calls the payloadCreator, dispatches pending, waits for resolution, dispatches fulfilled with result or rejected with error. In extraReducers, you handle these three cases with builder.addCase(). This eliminates the manual loading/error/success action pattern.

Q: What is reselect and when do you need it?

reselect's createSelector creates memoized selector functions. Problem: state => state.users.filter(u => u.active) creates a new array on every call → useSelector always sees a new reference → component always re-renders. createSelector takes input selectors and a result function — it only re-runs the result function if input values changed. Essential for any derived/computed state (filtered lists, sorted data, aggregations) used in useSelector.

Redux Saga Questions

Q: Why does Redux Saga use generator functions?

Three reasons: (1) Testability — yield produces plain effect descriptors. You can unit test by calling .next() and asserting effects without running actual APIs or timers. (2) Cancellability — generators can be cancelled mid-execution; async/await functions cannot be interrupted once started. (3) Control inversion — when you yield call(fn), the middleware decides when and how to call fn. This enables mocking, retry, timeout logic at the middleware level.

Q: Difference between call and fork?

call(fn) is blocking — saga suspends at that line until fn resolves. Like await. Use for sequential dependent operations. fork(fn) is non-blocking — starts fn in background, saga continues to next line immediately. Forked task errors propagate to parent. Use for background tasks, parallel work, child sagas you want to supervise. spawn is like fork but errors/cancellations are isolated — parent won't be affected if spawned task fails.

Q: takeEvery vs takeLatest?

takeEvery: listens for action, spawns a new worker saga for EVERY occurrence, all run concurrently. Use for independent operations: logging, analytics, notifications where every action matters. takeLatest: listens for action, cancels any currently running worker, starts new worker. Use for search-as-you-type, form submissions where only the latest result matters. Rule: if order matters and you want deduplication → takeLatest. If all must complete → takeEvery.

Q: How do you cancel an API call in saga?

Three ways: (1) takeLatest automatically cancels previous saga when new action comes — if saga is mid-fetch, it's cancelled. (2) race() — yield race({ data: call(api.fetch), cancel: take('CANCEL_ACTION') }) — whichever happens first wins, the other is cancelled. (3) Manual: const task = yield fork(workerSaga); yield take('CANCEL'); yield cancel(task). For actual HTTP cancellation, use AbortController inside the API call function.

Q: How do you handle race conditions in Sagas?

race() is built exactly for this: yield race({ result: call(fetchData), timeout: delay(5000) }). The first one to complete wins; others are cancelled. For search race conditions (old response arrives after newer one), takeLatest is the solution — it automatically cancels previous in-flight requests. For explicit competition (user logs out mid-login), race({ loginResult: call(login), logout: take('LOGOUT') }).

Q: How do you test sagas without mocking APIs?

Sagas yield plain effect descriptors — call(api.fetch, id) returns { type: 'CALL', fn: api.fetch, args: [id] } — a plain object. You can call gen.next().value and deep-equal it against the expected effect object without ever running the API. Pass mock results back via gen.next(mockData). This is pure unit testing: fast, no network, no flakiness. For integration tests with real dispatching, use runSaga() helper.

Q: How do you debounce an action in Redux Saga?

Two approaches: (1) takeLatest + delay(): yield takeLatest('SEARCH', worker) where worker starts with yield delay(400). Rapid actions cancel previous worker before delay completes → only last one after 400ms silence fires the API. (2) Built-in debounce effect: yield debounce(400, 'SEARCH_INPUT', searchWorker) — same result, cleaner syntax. For throttling: yield throttle(300, 'SCROLL', scrollWorker).

Q: What is the Watcher/Worker pattern?

Watcher saga: uses takeEvery/takeLatest/takeLeading to listen for specific actions. Never does real work. Worker saga: receives the action, does the actual async work (call, put), handles try/catch. Root saga: combines all watchers via yield all([watchA(), watchB()]). This separation keeps code clean: watchers define WHEN to react, workers define HOW to react. Each feature gets its own watcher/worker pair in its own file.

Q: What happens if a saga throws an error that isn't caught?

An uncaught error in a forked saga propagates to the parent saga and causes it to error too — cascading up to root. If root saga crashes, the entire saga middleware stops — no more sagas run. This is why you always wrap worker sagas in try/catch, and use spawn (not fork) for independent tasks like analytics where failure shouldn't bring down the main saga. You can also use sagaMiddleware's onError option for global error reporting.
