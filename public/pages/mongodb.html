Core Concepts

Core

What is MongoDB?

MongoDB is a document-oriented NoSQL database that stores data in flexible, JSON-like documents (BSON format) instead of rows and columns. Built for scale, flexibility, and developer speed.

  • Released in 2009 by 10gen (now MongoDB Inc.)
  • Documents can have different fields — no fixed schema
  • Horizontally scalable via sharding
  • Supports ACID transactions (since v4.0)
Interview Answer "MongoDB is a document database that stores data as BSON documents inside collections. Unlike relational DBs, there's no fixed schema — each document can have different fields. It's designed for horizontal scaling, flexible data models, and high-throughput workloads."

Q: When would you choose MongoDB over PostgreSQL?

Choose MongoDB when: data is hierarchical/nested (product catalogs, user profiles), schema evolves rapidly, you need horizontal scaling from day one, or you're storing semi-structured data. Choose PostgreSQL for: financial transactions, complex multi-table joins, strict ACID requirements, or when data is truly relational.

Find — Query & Projection

Find Operations
// Find all
db.users.find({});

// Find with filter
db.users.find({ age: { $gte: 25, $lte: 35 } });

// Logical operators
db.users.find({
  $or: [{ city: "Delhi" }, { city: "Faridabad" }]
});
db.users.find({
  $and: [{ age: { $gt: 20 } }, { active: true }]
});

// Projection — include/exclude fields
db.users.find({}, { name: 1, email: 1, _id: 0 }); // show only name+email
db.users.find({}, { password: 0 });              // hide password

// Querying nested fields
db.users.find({ "address.city": "Faridabad" }); // dot notation

// Array queries
db.users.find({ skills: "React Native" });          // contains element
db.users.find({ skills: { $all: ["React", "Node"] } }); // contains all
db.users.find({ skills: { $size: 3 } });           // array length = 3

// Sort, Skip, Limit
db.users.find({}).sort({ age: -1 }).skip(10).limit(5); // pagination

// findOne — returns first match
db.users.findOne({ email: "shailesh@example.com" });

Update

Update Operations
// updateOne — first matching document
db.users.updateOne(
  { _id: ObjectId("...") },              // filter
  { $set: { role: "Senior Dev" } }         // update
);

// updateMany — all matching documents
db.users.updateMany(
  { company: "PB Fintech" },
  { $set: { active: true } }
);

// replaceOne — replaces entire document (keeps _id)
db.users.replaceOne(
  { _id: ObjectId("...") },
  { name: "New Doc", role: "Admin" }
);

// Update operators
db.users.updateOne({ _id: id }, {
  $set:   { name: "Updated" },         // set field
  $unset: { oldField: "" },             // remove field
  $inc:   { loginCount: 1 },            // increment
  $push:  { tags: "mongodb" },          // add to array
  $pull:  { tags: "old-tag" },          // remove from array
  $addToSet: { skills: "GraphQL" }      // add if not exists
});

// UPSERT — insert if not found, update if found
db.users.updateOne(
  { email: "new@example.com" },
  { $set: { name: "New User", active: true } },
  { upsert: true }  // ← key option
);
// Returns: matchedCount: 0, upsertedId: ObjectId(...) if new
// Returns: matchedCount: 1, modifiedCount: 1 if existing

// findOneAndUpdate — returns document (before or after)
db.users.findOneAndUpdate(
  { email: "shailesh@example.com" },
  { $inc: { points: 10 } },
  { returnDocument: 'after' }  // return updated doc
);
$set vs replaceOne Trap updateOne with $set only updates specified fields. replaceOne replaces the ENTIRE document (only _id is kept). Accidentally using replaceOne when you meant $set will delete all other fields!

Delete

Delete Operations
// Delete one
db.users.deleteOne({ _id: ObjectId("...") });

// Delete many
db.users.deleteMany({ active: false });

// Delete all documents (collection stays)
db.users.deleteMany({});

// Drop collection entirely
db.users.drop();

// findOneAndDelete — returns deleted document
const deleted = db.users.findOneAndDelete({ _id: id });

Indexing — Deep Dive

🔥 Most Asked

What is an Index?

An index is a separate data structure that stores a small portion of a collection's data in an easy-to-traverse form. Without an index, MongoDB must scan every document (COLLSCAN). With an index, it jumps directly to matches (IXSCAN).

Creating Indexes
// Single field index
db.users.createIndex({ email: 1 });          // 1 = ascending
db.users.createIndex({ age: -1 });           // -1 = descending

// Unique index
db.users.createIndex({ email: 1 }, { unique: true });

// View all indexes
db.users.getIndexes();

// Drop index
db.users.dropIndex("email_1");

How Indexing Works Internally — B+ Tree

MongoDB indexes are implemented as B+ Trees (Balanced Plus Trees). Here's why this structure is perfect for databases:

  • Balanced — all leaf nodes are at the same depth → O(log n) lookups always
  • Sorted — data in leaf nodes is sorted → range queries are fast
  • Linked leaves — leaf nodes are linked → sequential scan without going back to root
  • High fan-out — internal nodes store many keys → shallow tree → fewer disk reads

Each leaf → pointer to actual document on disk
Q: Why B+ Tree and not B-Tree or Hash index?

B+ Tree over B-Tree: all data in leaves (not internal nodes), so sequential range scans traverse linked leaf list without backtracking. Hash index: O(1) equality lookup but CANNOT do range queries ({$gt/$lt}) or sort. B+ Tree handles both. MongoDB also supports hashed indexes but only for equality.

// Always use explain() to check query plan
db.users.find({ email: "test@test.com" }).explain("executionStats");

// Key fields to look at in explain output:
{
  queryPlanner: {
    winningPlan: {
      stage: "IXSCAN",          // ✅ good — using index
      // OR stage: "COLLSCAN"    ❌ bad — full scan
      indexName: "email_1"
    }

  executionStats: {
    nReturned: 1,
    totalDocsExamined: 1,       // ✅ examined == returned (perfect)
    totalKeysExamined: 1,
    executionTimeMillis: 0
    // If totalDocsExamined >> nReturned → index not selective enough
  }
}

Covered Queries — Zero Document Reads

A covered query is one where the index contains ALL the fields needed — MongoDB never has to touch the actual documents. This is the fastest possible query.

Covered Query Example
// Create compound index covering name and email
db.users.createIndex({ name: 1, email: 1 });

// This query is COVERED — filter + projection both in index
db.users
  .find({ name: "Shailesh" }, { name: 1, email: 1, _id: 0 })
  .explain();

// explain() shows: "stage": "PROJECTION_COVERED"
// totalDocsExamined: 0  ← zero document reads!

// NOT covered if you include a field not in the index:
db.users.find({ name: "Shailesh" }, { name: 1, age: 1 }); // age not in index
Senior-Level Tip Always exclude _id (set _id: 0) in projection when trying to get covered queries — _id is always returned by default and it IS in the index, but if you have custom projections, explicitly exclude it to keep the query covered.

Index Types & Cardinality

Index Cardinality = number of distinct values. High cardinality (email, userId) → very selective → great indexes. Low cardinality (boolean, status with 3 values) → not selective → often not worth indexing.

Compound Index — Order Is Critical
// Compound index
db.orders.createIndex({ status: 1, userId: 1, createdAt: -1 });

// ESR Rule: Equality → Sort → Range
// Put equality fields first, sort fields next, range fields last

// ✅ Uses index fully (prefix match: status + userId)
db.orders.find({ status: "active", userId: "u123" });

// ✅ Uses first part of index (prefix: status)
db.orders.find({ status: "active" });

// ❌ CANNOT use this compound index (skips status)
db.orders.find({ userId: "u123" }); // needs separate index

// Index prefix rule: {a,b,c} index supports queries on:
// a | a,b | a,b,c — but NOT b | c | b,c alone
Q: What is the ESR Rule for compound indexes?

Equality → Sort → Range. Put fields used in equality matches first, fields used for sorting second, fields used in range queries (gt, lt, in) last. This ordering maximizes index utilization and avoids in-memory sorts.

Multikey Index — Arrays
// Multikey index — automatically created when indexing array field
db.products.createIndex({ tags: 1 });

// MongoDB creates one index entry PER array element
// { tags: ["mobile", "fintech", "insurance"] }
// Creates 3 index entries: "mobile", "fintech", "insurance"

// ✅ Fast query
db.products.find({ tags: "fintech" });

// ⚠ Compound multikey: max ONE array field per compound index
db.col.createIndex({ arr1: 1, arr2: 1 }); // ❌ error if both are arrays

When Indexes Are NOT Used

  • $regex without anchoring — { name: /shailesh/ } may COLLSCAN. Use /^shailesh/ (anchored) to use index.
  • Negation operators — $ne, $nin, $not often cannot use index efficiently
  • JavaScript expressions — $where bypasses indexes entirely
  • Low cardinality + high result % — if query returns >30% of documents, MongoDB may choose COLLSCAN anyway
  • Field used in expression — { $expr: { $gt: ["$field", 5] } } may not use index
The Tradeoff Every index speeds up reads but slows down writes. Each insert/update/delete must also update ALL relevant indexes. A collection with 10 indexes has 10x write overhead vs no indexes. Index only what you query frequently.

Aggregation Framework

🔥 Interview Core

Aggregation Pipeline

The aggregation pipeline processes documents through a series of stages. Each stage transforms the documents and passes them to the next. Think of it as a Unix pipe: data flows through transformations.

Real-World Pipeline — Sales Report
// "Get top 5 products by revenue for active orders in 2024"
db.orders.aggregate([
  // Stage 1: $match — filter early (like WHERE + uses indexes)
  {
    $match: {
      status: "completed",
      createdAt: { $gte: ISODate("2024-01-01") }
    }
  },

  // Stage 2: $unwind — flatten array field into separate docs
  { $unwind: "$items" },

  // Stage 3: $group — like SQL GROUP BY
  {
    $group: {
      _id: "$items.productId",       // group by product
      totalRevenue: { $sum: { $multiply: ["$items.price", "$items.qty"] } },
      totalOrders: { $sum: 1 },
      avgPrice: { $avg: "$items.price" }
    }
  },

  // Stage 4: $project — reshape output (like SELECT)
  {
    $project: {
      productId: "$_id",
      totalRevenue: 1,
      totalOrders: 1,
      avgPrice: { $round: ["$avgPrice", 2] },
      _id: 0
    }
  },

  // Stage 5: $sort — sort by revenue desc
  { $sort: { totalRevenue: -1 } },

  // Stage 6: $limit — top 5
  { $limit: 5 }
]);

$lookup — MongoDB JOINs

$lookup — Left Outer Join
// Join orders with users collection
db.orders.aggregate([
  { $match: { status: "pending" } },
  {
    $lookup: {
      from: "users",             // join with this collection
      localField: "userId",     // field in orders
      foreignField: "_id",       // field in users
      as: "userDetails"         // output array name
    }
  },
  // userDetails is an array — unwind to get single object
  { $unwind: { path: "$userDetails", preserveNullAndEmptyArrays: true } },
  {
    $project: {
      orderId: "$_id",
      amount: 1,
      userName: "$userDetails.name",
      userEmail: "$userDetails.email"
    }
  }
]);

// Advanced $lookup with pipeline (MongoDB 3.6+)
{
  $lookup: {
    from: "products",
    let: { productIds: "$items.productId" },
    pipeline: [
      { $match: { $expr: { $in: ["$_id", "$$productIds"] } } },
      { $project: { name: 1, price: 1 } }
    ],
    as: "productDetails"
  }
}
$lookup Performance Warning $lookup is expensive — it performs a join in-memory. Always $match before $lookup to reduce documents. Add indexes on the foreignField in the joined collection. For high-performance systems, consider denormalizing data instead of using $lookup.

Embedding vs Referencing Examples
// ✅ EMBED — address is always fetched with user
{
  _id: ObjectId("..."),
  name: "Shailesh",
  address: {                  // embedded
    street: "Sector 14",
    city: "Faridabad"
  }
}

// ✅ EMBED — order items are part of the order
{
  _id: ObjectId("..."),
  orderId: "ORD-001",
  items: [                    // embedded array
    { productId: "P1", qty: 2, price: 500 }
  ]
}

// ✅ REFERENCE — comments can grow unbounded
// Post document:
{ _id: ObjectId("post1"), title: "React Native Tips" }

// Comment document (separate collection):
{ postId: ObjectId("post1"), text: "Great post!", author: "Bob" }

// ✅ REFERENCE — users shared across many entities
{ _id: ObjectId("..."), title: "Policy 101", ownerId: ObjectId("user1") }

Schema Design Rules of Thumb

  • Embed when: data is read together, one-to-one or one-to-few, child data doesn't need to be accessed independently
  • Reference when: many-to-many relationships, large arrays that grow unbounded, data shared across many documents, child data is accessed independently
  • 16MB BSON limit: if embedded arrays can grow unbounded (comments, logs, events), always reference
  • Access pattern drives design: design schema around how you query, not how data is naturally related
Q: Design a schema for a blog with posts and comments

Posts collection with embedded author info (snapshot — denormalized for read speed). Comments in a separate collection referencing postId — comments can grow unbounded, users may want to query "all comments by user", and real-time updates to comments shouldn't require loading the whole post. Use $lookup or application-level join when rendering full post with comments.

Multi-Document Transactions

Transaction — Bank Transfer
const session = client.startSession();

try {
  session.startTransaction({
    readConcern: { level: 'snapshot' },
    writeConcern: { w: 'majority' }
  });

  // Debit sender
  await db.accounts.updateOne(
    { userId: "sender", balance: { $gte: 1000 } },
    { $inc: { balance: -1000 } },
    { session }
  );

  // Credit receiver
  await db.accounts.updateOne(
    { userId: "receiver" },
    { $inc: { balance: 1000 } },
    { session }
  );

  // Log transaction
  await db.txnLog.insertOne(
    { from: "sender", to: "receiver", amount: 1000, date: new Date() },
    { session }
  );

  await session.commitTransaction();
} catch (err) {
  await session.abortTransaction(); // ← all changes rolled back
  throw err;
} finally {
  session.endSession();
}
Transaction Cost Multi-document transactions have significant performance overhead (locking, coordination across replica set). Use them only when necessary (financial ops, critical multi-step operations). Prefer single-document atomic operations when possible via embedding.
Q: What is write concern and read concern?

Write concern {w: 'majority'} = wait until majority of replica set members acknowledge the write. More durable but slower. Read concern 'snapshot' = reads data from a consistent snapshot across the transaction. 'local' = reads from primary without guaranteeing it was replicated. For financial ops, always use majority write concern.

Core

Replica Sets

A replica set is a group of MongoDB instances that maintain the same dataset. Provides redundancy, high availability, and read scaling.

REPLICA SET (3 MEMBERS)
PRIMARY
Read + Write
→ oplog →
SECONDARY 1
Read (optional)
→ oplog →
SECONDARY 2
Read (optional)
If Primary fails → automatic election → new Primary in seconds
  • Primary — receives all writes. Logs operations to oplog (operations log)
  • Secondary — replicates from Primary's oplog asynchronously. Can serve reads
  • Arbiter — participates in elections only, stores no data. Used for tie-breaking
  • Minimum recommended: 3 members (1P + 2S) for majority election
Q: What is an oplog and how does replication work?

The oplog (operations log) is a special capped collection on the Primary that records all write operations in idempotent form. Secondaries tail the oplog and replay operations to stay in sync. Because it's capped, old entries roll off — if a Secondary falls too far behind, it needs a full resync (initial sync).

Q: What happens during a failover?

1) Primary becomes unreachable. 2) Secondaries detect this via heartbeats (every 2s). 3) An election is triggered — each member votes (based on priority, oplog position). 4) Member with majority votes becomes new Primary. 5) Clients reconnect automatically (driver handles this). Typical downtime: 10-30 seconds.

Sharding — Very Deep

🔥 Senior Level

What is Sharding & Why Use It?

Sharding is MongoDB's method for horizontal scaling — distributing data across multiple servers (shards). When a single machine can't handle the data volume or throughput, you shard.

  • Volume scaling: 10TB collection? Split across 10 shards = 1TB each
  • Write scaling: distribute writes across multiple primaries
  • Read scaling: targeted queries go to one shard, not all
Sharding vs Replication Replication = same data on multiple nodes (availability + redundancy). Sharding = different data on different nodes (capacity + throughput). Production: you use BOTH — each shard is itself a replica set.

Shard Key — The Most Critical Decision

The shard key determines HOW data is distributed across shards. It is immutable — cannot be changed without resharding. A bad shard key will ruin performance forever.

Enabling Sharding
// Enable sharding on database
sh.enableSharding("policybazaar");

// Create index on shard key (required first)
db.policies.createIndex({ userId: 1 });

// Shard the collection
sh.shardCollection("policybazaar.policies", { userId: 1 });

// Check sharding status
sh.status();
Q: What makes a GOOD shard key?

Good shard key has: (1) High cardinality — enough distinct values to create many chunks. (2) Low frequency — no single value dominates (avoid "hot shard"). (3) Non-monotonically increasing — auto-incrementing IDs or timestamps create "hot spot" on one shard as all new writes go there. (4) Query targeted — if most queries filter by it, queries go to one shard not all.

// Hash sharding — distributes writes uniformly
sh.shardCollection("db.events", { userId: "hashed" });

// MongoDB hashes the userId value → uniform distribution
// "user123" → hash → 7823647 → goes to shard 2
// "user124" → hash → 1293847 → goes to shard 1  (random!)

Scatter-Gather Problem

When a query does NOT include the shard key, mongos must broadcast it to ALL shards and gather results. This is called scatter-gather and is very expensive at scale.

Targeted vs Scatter-Gather
// Shard key: { userId: 1 }

// ✅ TARGETED — mongos routes to exactly ONE shard
db.policies.find({ userId: "user123" });

// ✅ TARGETED — shard key in query
db.policies.find({ userId: "user123", status: "active" });

// ❌ SCATTER-GATHER — no shard key → all shards queried
db.policies.find({ status: "active" }); // goes to ALL shards
db.policies.find({ premium: { $gt: 5000 } }); // goes to ALL shards
Q: Walk me through the query routing flow in a sharded cluster

1) App connects to mongos (router). 2) mongos parses the query — checks if shard key is in the filter. 3) If yes: reads chunk metadata from config servers cache → identifies which shard(s) own those ranges → routes to 1-2 shards. 4) If no: scatter-gather — sends query to ALL shards, waits for all responses, merges results, returns to client. 5) mongos handles sorting/limiting across merged results.

Chunks & Rebalancing

  • Data is divided into chunks — contiguous ranges of shard key values (default: 128MB)
  • When a chunk exceeds the limit, it splits into two smaller chunks
  • The balancer (runs on config server) monitors chunk distribution and migrates chunks between shards to keep counts balanced
  • Chunk migration impacts performance — schedule during off-peak hours
Interview Gold — Jumbo Chunks A "jumbo chunk" cannot be split — all documents have the same shard key value. Example: if status is the shard key and 80% of docs have status="active", that chunk is jumbo and can't be distributed. This is why low-cardinality shard keys are dangerous.

const result = db.orders.find({ userId: "u123" }).explain("executionStats");

// Key metrics to check:
result.executionStats = {
  nReturned: 5,             // documents returned
  totalDocsExamined: 5,     // ✅ should equal nReturned
  totalKeysExamined: 5,     // ✅ should equal nReturned
  executionTimeMillis: 2    // ✅ fast
}

// ❌ RED FLAG: nReturned=5 but totalDocsExamined=100000
// → no index, or index not selective enough

// 3. allPlansExecution — runs all candidate plans
db.orders.find({...}).explain("allPlansExecution");

Performance Checklist

  • Index on query fields — every field in $match, $sort should be indexed
  • $match early in pipeline — filter before $lookup, $unwind, $group
  • Limit projection — only fetch needed fields (bandwidth savings)
  • Covered queries — include projected fields in index
  • Avoid $where — JS in queries is slow, bypasses indexes
  • Use $exists sparingly — if checking a field exists on most docs, better to set a default value
  • Partial indexes — index only subset of documents (e.g., only active users)
Partial Index — Index Only Active Users
// Only index active users — smaller index, faster for active-user queries
db.users.createIndex(
  { email: 1 },
  { partialFilterExpression: { active: true } }
);

// ✅ Uses partial index
db.users.find({ email: "x@y.com", active: true });

// ❌ DOES NOT use partial index (active not specified)
db.users.find({ email: "x@y.com" });

Concurrency — Document-Level Locking

MongoDB uses document-level locking (not table or database level). Multiple operations can run concurrently on different documents in the same collection.

  • WiredTiger storage engine (default since MongoDB 3.2) uses document-level locking
  • Multiple readers can access same document simultaneously
  • Writers get exclusive lock on document only
  • Much better concurrency than MySQL table-level locks for high write workloads
Q: How does MongoDB prevent dirty reads in transactions?

MongoDB uses MVCC (Multi-Version Concurrency Control) — reads see a consistent snapshot of data at the start of the transaction. Uncommitted writes from other transactions are never visible. This is snapshot isolation, preventing dirty reads and non-repeatable reads.

🔥 System Design

Scenario 1: Slow Query Debugging

Interviewer says: "A MongoDB query is taking 10 seconds. Walk me through debugging it."
  1. Run explain("executionStats") on the query — check for COLLSCAN vs IXSCAN, totalDocsExamined vs nReturned ratio
  2. Check for missing index — if COLLSCAN on large collection, create index on filter fields
  3. Check query selectivity — if index exists but ratio is bad, the index may have low cardinality
  4. Check compound index order — ESR rule? Is the first field in index actually being used?
  5. Check if $regex is anchored — unanchored regex can't use index
  6. Check aggregation pipeline — is $match first? Is $lookup before $match (huge mistake)?
  7. Check for in-memory sort — explain shows SORT stage? Add sort field to index
  8. Check document size — very large documents slow everything down even with indexes
Slow Query Debugging Workflow
// Step 1: Run explain
db.orders.find({ status: "active", userId: "u123" })
  .sort({ createdAt: -1 })
  .explain("executionStats");

// Step 2: Look for problems:
// - stage: "COLLSCAN" → missing index
// - stage: "SORT" → no index supporting sort (in-memory sort)
// - totalDocsExamined >> nReturned → poor selectivity
// - multiple FETCH stages → multiple random reads

// Step 3: Create proper compound index (ESR rule)
// E: userId (equality), R: createdAt (range/sort)
db.orders.createIndex({ userId: 1, status: 1, createdAt: -1 });

// Step 4: Re-run explain — confirm IXSCAN + no in-memory SORT

Scenario 3: SQL vs MongoDB Decision

Q: You're building a policy management system for Policybazaar. MongoDB or PostgreSQL?

This is nuanced. Use PostgreSQL for: core policy data with strict financial accuracy, premium calculations, multi-table joins across policies/users/payments, complex reporting. Rapid Fire Interview Q&A

Q: What is the maximum document size in MongoDB?

16MB BSON document limit. For larger data (images, PDFs), use GridFS which splits files into 255KB chunks stored in two collections: fs.files (metadata) and fs.chunks (data).

Q: What is the _id field and can you customize it?

_id is the primary key — auto-generated as ObjectId if not specified. You CAN provide your own _id (any unique value — string, number, UUID). The _id is always indexed. Avoid using natural data (email, username) as _id — they change and cascading updates are painful.

Q: Difference between $push and $addToSet?

$push always adds the element (allows duplicates). $addToSet only adds if the element doesn't already exist (like a set). Use $addToSet for tags, roles, unique array values. Use $push for ordered arrays where duplicates are valid (chat messages, history logs).

Q: What is an atlas search? When would you use it over regular queries?

Atlas Search is MongoDB's full-text search built on Apache Lucene. Use for: fuzzy matching ("shailesh" matches "shailesh"), relevance ranking, autocomplete, language-aware search (stemming, stop words). Regular indexes for exact/range matches. Atlas Search for Google-like search experiences.

Q: How do you handle schema migration in MongoDB?

MongoDB's flexible schema means migration is different from SQL. Options: (1) Lazy migration — update schema in app code, handle both old/new format, backfill gradually. (2) Bulk update — db.collection.updateMany({}, [migration pipeline]). (3) Use MongoDB schema versioning pattern — add a schemaVersion field, handle in app. No ALTER TABLE needed.

Q: What is Read Preference in MongoDB?

Read preference controls where reads go in a replica set. Primary (default): all reads from primary. PrimaryPreferred: primary if available, else secondary. Secondary: always from secondary. SecondaryPreferred: secondary if available. Nearest: lowest latency member. Use Secondary/SecondaryPreferred for analytics to offload primary.

Q: Can you update the shard key value of a document?

Yes, since MongoDB 4.2 you can update shard key values (except _id as shard key). However, it's a multi-document transaction internally and may require a document to move between shards (expensive). Choose a shard key that rarely needs updating. In older versions, you had to delete and re-insert.
