How Browsers Parse & Render HTML

🔥 Most Asked

Critical Rendering Path — Step by Step

Understanding this is what separates mid-level from senior developers. Every HTML page goes through these exact steps:

  1. Bytes → Characters — browser decodes raw bytes into characters using charset (UTF-8)
  2. Characters → Tokens — tokenizer converts text into HTML tokens (<html>, <body>, text nodes)
  3. Tokens → Nodes — each token becomes a node object with properties
  4. Nodes → DOM — nodes linked into the Document Object Model tree
  5. CSS → CSSOM — same process for stylesheets: bytes → CSSOM tree
  6. DOM + CSSOM → Render Tree — only visible elements (no display:none, no <head>)
  7. Layout (Reflow) — calculate exact position and size of every element
  8. Paint — fill in pixels for each element (color, text, images)
  9. Composite — layer painted sections, GPU handles transforms/opacity
Interview Gold Layout → Paint → Composite. Animations using transform and opacity skip Layout and Paint (GPU composited) — this is why they're 60fps. Animations using width/height/top/left trigger full Layout recalculation — expensive!
Q: What is "reflow" and "repaint"?

Reflow (Layout): recalculates geometry — position, size — of affected elements and all their children. Triggered by: changing width/height/margin/padding, adding/removing elements, changing font size, window resize. Repaint: redraws pixels without changing geometry — triggered by color, background, visibility changes. Reflow is more expensive because it always causes repaint, but repaint doesn't always cause reflow.

DOCTYPE, Head & Meta Tags

Core

DOCTYPE — Why It Matters

Modern vs Legacy DOCTYPE
<!-- HTML5 DOCTYPE — always use this -->
<!DOCTYPE html>

<!-- Legacy HTML4 (never write this) -->
<!DOCTYPE HTML PUBLIC "-//W3C//DTD HTML 4.01//EN"
   "http://www.w3.org/TR/html4/strict.dtd">

Without DOCTYPE, browsers enter Quirks Mode — they emulate old IE behaviour with broken box model and inconsistent rendering. DOCTYPE is not an HTML tag — it's a declaration telling the browser which HTML version to expect.

Q: What happens without DOCTYPE? What is Quirks Mode?

Without DOCTYPE, browser enters Quirks Mode — emulates bugs from IE5. Box model is broken: width/height include padding and border (opposite of CSS standard). Many CSS properties behave differently. Always include <!DOCTYPE html> as the very first line — nothing, not even a space, should come before it.

Essential Head Tags

Complete Head Template
<head>
  <!-- Character encoding — MUST be first -->
  <meta charset="UTF-8">

  <!-- Viewport — essential for responsive design -->
  <meta name="viewport" content="width=device-width, initial-scale=1.0">

  <!-- Page title — shown in tab + SEO -->
  <title>Page Title — Site Name</title>

  <!-- SEO description -->
  <meta name="description" content="150-160 char description for search results">

  <!-- Robots control -->
  <meta name="robots" content="index, follow">

  <!-- Canonical URL — prevent duplicate content -->
  <link rel="canonical" href="https://example.com/page">

  <!-- Open Graph -->
  <meta property="og:title"       content="...">
  <meta property="og:description"  content="...">
  <meta property="og:image"        content="https://...">
  <meta property="og:url"          content="https://...">

  <!-- Favicon -->
  <link rel="icon" type="image/svg+xml" href="/favicon.svg">

  <!-- CSS — render-blocking, put in head -->
  <link rel="stylesheet" href="styles.css">

  <!-- Preconnect for external resources -->
  <link rel="preconnect" href="https://fonts.googleapis.com">
</head>
Charset Must Be First <meta charset="UTF-8"> must appear within the first 1024 bytes of the HTML file. If the browser encounters characters before knowing the charset, it may misinterpret them (especially for non-ASCII characters). Place it as the very first child of <head>.

Semantic HTML

🔥 Most Asked

What is Semantic HTML & Why Does It Matter?

Semantic HTML uses elements that describe their meaning, not just their appearance. A <div> means nothing. A <nav> means "this is navigation". Three stakeholders benefit:

  • Accessibility — screen readers announce elements correctly ("navigation landmark", "main content")
  • SEO — search engines understand page structure and content hierarchy
  • Developers — code is self-documenting and easier to maintain
Non-semantic vs Semantic
<!-- ❌ Non-semantic — div soup -->
<div class="header">
  <div class="nav">
    <div class="nav-link">Home</div>
  </div>
</div>
<div class="main">
  <div class="article">
    <div class="heading">Title</div>
  </div>
</div>

<!-- ✅ Semantic — meaningful structure -->
<header>
  <nav>
    <a href="/">Home</a>
  </nav>
</header>
<main>
  <article>
    <h1>Title</h1>
  </article>
</main>

Q: What is the difference between article, section, and div?

article: self-contained content that makes sense alone and could be syndicated (blog post, tweet, product). If you can take it out and it still makes sense — it's an article. section: thematic grouping that belongs to a larger whole — chapters, tab panels, steps in a guide. Always needs a heading. div: absolutely no semantic meaning — pure styling/scripting container. Rule: reach for semantic elements first; use div only when nothing semantic fits.

Q: Can you use header and footer multiple times?

Yes! header and footer are scoped to their nearest sectioning element (article, section, aside, nav). A page can have a main header/footer AND individual article header/footers. Only main is restricted to one per page.

Heading Hierarchy — Critical for Accessibility & SEO

Correct Heading Hierarchy
<!-- ✅ Logical hierarchy -- don't skip levels -->
<h1>Insurance Guide</h1>        <!-- one per page -->
  <h2>Term Insurance</h2>
    <h3>Benefits</h3>
    <h3>How to Apply</h3>
  <h2>Health Insurance</h2>

<!-- ❌ Wrong -- skipped h2 -->
<h1>Title</h1>
<h3>Subtitle</h3>  <!-- screen readers confused -->

<!-- ❌ Wrong -- multiple h1 (bad for SEO) -->
<h1>First Heading</h1>
<h1>Second Heading</h1>

Links & Navigation

Core

anchor <a> — All Attributes

Anchor Element Complete Reference
<!-- Basic link -->
<a href="https://example.com">External Link</a>

<!-- Open in new tab -->
<a href="https://example.com"
   target="_blank"
   rel="noopener noreferrer">New Tab</a>
<!-- noopener: prevents new tab accessing window.opener (security)
     noreferrer: doesn't send Referer header + implies noopener -->

<!-- Download link -->
<a href="/report.pdf" download="Annual-Report-2024.pdf">Download</a>

<!-- Email / phone / SMS -->
<a href="mailto:hello@example.com?subject=Hello">Email</a>
<a href="tel:+919876543210">Call Us</a>
<a href="sms:+919876543210">SMS</a>

<!-- Jump to section (in-page navigation) -->
<a href="#section-id">Jump to Section</a>

<!-- Skip link (accessibility) -->
<a href="#main-content" class="skip-link">Skip to main content</a>
Q: Why use rel="noopener noreferrer" on target="_blank"?

Security: without noopener, the new tab can access the original page via window.opener and redirect it to a malicious URL (reverse tabnapping attack). noopener sets window.opener to null. noreferrer additionally prevents the Referer header being sent (privacy) and implies noopener. Always add both when opening external links in new tabs.

Images & Media

🔥 Most Asked

img — All Critical Attributes

Image Best Practices
<!-- Always include alt (accessibility + SEO) -->
<img src="hero.jpg"
     alt="Shailesh presenting at tech conference"
     width="800"
     height="450"
     loading="lazy"
     decoding="async">

<!-- Decorative image — empty alt (screen reader skips it) -->
<img src="divider.svg" alt="" role="presentation">

<!-- Responsive images with srcset -->
<img
  src="image-800.jpg"
  srcset="image-400.jpg 400w,
          image-800.jpg 800w,
          image-1200.jpg 1200w"
  sizes="(max-width: 600px) 400px,
         (max-width: 1000px) 800px,
         1200px"
  alt="Responsive image">

<!-- picture — art direction (different crops) + format fallback -->
<picture>
  <source type="image/avif" srcset="hero.avif">
  <source type="image/webp" srcset="hero.webp">
  <source media="(max-width: 600px)" srcset="hero-mobile.jpg">
  <img src="hero.jpg" alt="Hero image">  <!-- fallback -->
</picture>
Q: Why set width and height attributes on images?

Setting explicit width/height on <img> allows the browser to reserve space before the image loads, preventing Cumulative Layout Shift (CLS) — a Core Web Vitals metric. Without dimensions, page content jumps as images load. The browser calculates the aspect ratio from the attributes and holds that space. You can still override with CSS — the attribute values just define the aspect ratio hint.

Q: Difference between srcset and picture?

srcset: same image in different resolutions/sizes — browser picks the best based on screen density and viewport. Good for resolution switching. picture: different images for different contexts — art direction (mobile crop vs desktop), format selection (avif → webp → jpg). Use srcset for "same image, different sizes". Use picture when you need completely different images or format fallbacks.

Q: What is the difference between button type="submit", type="button", and type="reset"?

type="submit" (default): submits the form. type="button": does nothing by default — needs JavaScript onclick. type="reset": resets all inputs to default values (rarely useful, never use in critical flows — users accidentally reset forms). Important: always specify type on button — without it, a button inside a form defaults to type="submit" and accidentally submits the form.

Intersection Observer API

Intersection Observer — Lazy Load / Infinite Scroll
const observer = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      // Element is visible in viewport
      entry.target.src = entry.target.dataset.src; // lazy load image
      observer.unobserve(entry.target); // stop watching
    }
  });
}, {
  root: null,         // null = viewport
  rootMargin: '200px', // trigger 200px before visible
  threshold: 0.1       // 10% visible = trigger
});

document.querySelectorAll('img[data-src]').forEach(img => observer.observe(img));

// Better than scroll events — no performance hit, browser-optimized

MutationObserver API

MutationObserver — Watch DOM Changes
const observer = new MutationObserver((mutations) => {
  mutations.forEach(mutation => {
    if (mutation.type === 'childList') {
      console.log('Children added/removed', mutation.addedNodes);
    }
    if (mutation.type === 'attributes') {
      console.log('Attribute changed:', mutation.attributeName);
    }
  });
});

observer.observe(targetElement, {
  childList: true,    // watch child add/remove
  attributes: true,   // watch attribute changes
  subtree: true,       // watch all descendants
  characterData: true  // watch text content
});
observer.disconnect(); // cleanup

Q: Why shouldn't you store auth tokens in localStorage?

localStorage is accessible by ANY JavaScript on the page — including injected third-party scripts. If the page has XSS vulnerability, tokens can be stolen. Cookies with HttpOnly flag are NOT accessible by JavaScript at all — even if XSS injects code, it can't read HttpOnly cookies. For auth tokens: use HttpOnly, Secure, SameSite=Strict cookies. For non-sensitive user preferences (theme, language): localStorage is fine.

Web Workers

Advanced

Running JS Off the Main Thread

JavaScript is single-threaded. Expensive computation blocks the UI. Web Workers run in a separate thread — no DOM access, communicate via message passing.

Web Worker Example
// main.js — create worker
const worker = new Worker('worker.js');

worker.postMessage({ data: largeArray }); // send to worker

worker.onmessage = (e) => {
  console.log('Result:', e.data); // receive from worker
};
worker.onerror = (e) => console.error(e);
worker.terminate(); // stop worker when done

// worker.js — runs in separate thread
self.onmessage = (e) => {
  const result = e.data.data.reduce((sum, n) => sum + n, 0); // heavy compute
  self.postMessage(result); // send back to main
};
// Workers can: fetch, use IndexedDB, import scripts
// Workers CANNOT: access DOM, window, document
Q: Difference between Web Workers, Service Workers, and Shared Workers?

Web Worker: one tab, dedicated to that page, dies with page. Service Worker: intercepts network requests, enables offline/PWA, runs independently of any tab (background). Shared Worker: shared across multiple tabs from same origin. Use Web Workers for CPU-intensive computation, Service Workers for caching/offline/push notifications, Shared Workers for cross-tab communication.

    super();

<!-- Usage -->

Security — XSS, CSRF, CSP

🔥 Senior Must-Know

XSS — Cross-Site Scripting

XSS: attacker injects malicious scripts into your page. When other users load the page, the script runs in their browser — can steal cookies, redirect, keylog.

XSS Prevention
// ❌ DANGEROUS — innerHTML with user content
element.innerHTML = userInput; // XSS if input = "<script>steal()</script>"

// ✅ SAFE — textContent (escapes HTML)
element.textContent = userInput;

// ✅ SAFE — createElement (never interprets as HTML)
const p = document.createElement('p');
p.textContent = userInput;

// ✅ SAFE — DOMPurify for rich content
element.innerHTML = DOMPurify.sanitize(userInput);

// CSP Header — whitelist what can run
// Content-Security-Policy: default-src 'self'; script-src 'self' cdn.example.com
<meta http-equiv="Content-Security-Policy"
      content="default-src 'self'; img-src https:; script-src 'self'">

Q: What is clickjacking and how do you prevent it?

Clickjacking: attacker loads your site in a transparent iframe on their site. User thinks they're clicking the attacker's content but actually clicking your site (like/buy/transfer buttons). Prevention: X-Frame-Options: DENY or SAMEORIGIN header prevents your page from being embedded in iframes. Or CSP: frame-ancestors 'none'. Also use framebusting JS as fallback.

Interview Q&A
26

Complete Interview Q&A — 2-3 Year Level

100% Prepare

Rendering & Browser

Q: What is the difference between DOMContentLoaded and load events?

DOMContentLoaded fires when HTML is fully parsed and DOM is ready — stylesheets, images, subframes may still be loading. This is when most JS that manipulates DOM should run. load fires when EVERYTHING is loaded: images, stylesheets, scripts, iframes. Use DOMContentLoaded for UI initialization. Use load for things that need full page including images (e.g., measuring image dimensions).

Q: What is the difference between window.onload and document.addEventListener('DOMContentLoaded')?

window.onload = DOMContentLoaded fires earlier, covers only HTML parsing. window.onload fires after all resources load. Also: onload can only have ONE handler (it's a property — last assignment wins). addEventListener allows multiple handlers. Always prefer addEventListener over onevent properties.

Q: What causes layout thrashing and how do you prevent it?

Layout thrashing: reading layout properties (offsetWidth, scrollTop, getBoundingClientRect) after writing styles in a loop. Each read forces the browser to recalculate layout synchronously. Fix: batch all reads before writes, or use requestAnimationFrame, or use FastDOM library. Example — wrong: for each el, setWidth(el) then readWidth(el). Right: read all first, then write all.

Q: What is the difference between innerHTML, textContent, and innerText?

innerHTML: gets/sets HTML including tags — XSS risk with user content, triggers parsing. textContent: gets/sets text only — no HTML parsing, all tags stripped, includes hidden elements, fastest. innerText: like textContent but CSS-aware — respects display:none (hidden text excluded), triggers reflow, slower. Use textContent for reading/writing plain text (safe), innerHTML only for trusted HTML content.

Q: How does the browser handle render-blocking resources?

CSS: always render-blocking — browser won't paint until CSSOM built. Minimize by inlining critical CSS, using media attributes for non-critical CSS, preloading. JS: parser-blocking by default — stops HTML parsing. Fix with async (independent scripts) or defer (ordered, after parsing). Fonts: can block text rendering — use font-display: swap in @font-face to show text immediately with fallback font.

HTML Elements & Semantics

Q: When would you use a button vs an anchor tag?

anchor (<a href>): navigates somewhere — different page, section, download. Has URL, can be right-clicked (open in new tab), bookmarked, shows in browser history. button: performs an action — submit form, toggle modal, trigger JS. No URL, no right-click menu. Rule: if it goes somewhere → anchor. If it does something → button. Never use <a href="#"> for actions — use button. Never use <div onclick> — keyboard users can't use it.

Q: What is the difference between strong, b, em, and i?

strong: semantic importance — content has strong importance, screen readers may add emphasis. b: stylistically bold without extra importance — product names, keywords. em: stress emphasis — changes meaning of sentence (I didn't say HE did it vs I didn't SAY he did it). i: alternate voice — technical terms, foreign words, thoughts. Use semantic elements (strong, em) when meaning matters; use CSS for purely visual styling.

Q: What are data attributes and when do you use them?

data-* attributes store custom data on HTML elements without affecting rendering. data-user-id="123", data-action="delete". Access via JS: element.dataset.userId (camelCase conversion auto). Use for: storing IDs for event delegation, passing config to widgets, communication between HTML and JS without class manipulation. Don't use for sensitive data — visible in HTML source.

Q: What is the difference between disabled and readonly on inputs?

disabled: visually grayed out, not focusable, not submitted with form, excluded from tab order, value ignored. readonly: can be focused, included in form submission, cannot be edited by user (but can be changed by JS). Use disabled for fields that shouldn't participate at all. Use readonly for displaying values that shouldn't be changed but must be submitted (like a confirmation of selected plan).

Q: What is the difference between <script>, <script async>, and <script defer>?

Normal: HTML parsing pauses, script downloads and executes, then parsing resumes. Async: downloads in parallel, executes immediately when done (interrupts parsing), no order guarantee. Defer: downloads in parallel, executes after HTML is fully parsed, in document order. For app code that needs DOM → defer. For independent analytics/ads → async. Both are better than placing scripts at end of body (they start downloading sooner).

Q: What is the purpose of the lang attribute on <html>?

<html lang="en"> tells browser and screen readers the language of the page. Screen readers use correct pronunciation and language-specific voice. Search engines use it for language targeting. Browsers apply correct hyphenation rules. Spell checkers use correct dictionary. Use language subtags for regional variants: lang="en-IN" (Indian English), lang="hi" (Hindi). Critical for internationalized apps.

Q: What are void elements / self-closing tags?

Void elements cannot have children and don't need a closing tag: <br>, <hr>, <img>, <input>, <link>, <meta>, <area>, <base>, <col>, <embed>, <source>, <track>, <wbr>. In HTML5, the slash is optional: <br> and <br /> are both valid. In JSX (React), the slash is required.

Forms & Inputs

Q: What is the difference between GET and POST in forms?

GET: data appended to URL as query string — visible in URL, cached, bookmarkable, can be shared, browser can prefetch, limited length (~2000 chars). Use for: search, filters, pagination. POST: data in request body — not in URL, not cached, more secure for sensitive data, no length limit. Use for: login, signup, payment, data creation/modification. Key point: GET is idempotent (same result repeatedly), POST is not.

Q: What is autocomplete on forms and when should you disable it?

autocomplete="on/off" controls browser autofill. Use specific values for best UX: autocomplete="email", "current-password", "new-password", "name", "tel", "address-line1", "cc-number". Disable for: OTP fields (autocomplete="one-time-code" instead), security-critical new passwords, fields that confuse autofill. Never disable for login forms — users rely on password managers (autocomplete="off" breaks them).

Q: How does form enctype work?

enctype defines how form data is encoded when submitted. application/x-www-form-urlencoded (default): key=value pairs URL-encoded, & separated — good for text data. multipart/form-data: sends each field as a separate part with boundary — REQUIRED for file uploads, slightly larger. text/plain: no encoding — only for debugging, never in production. In React: forms use fetch/axios so enctype matters less, but the underlying concept remains.

Performance & Core Web Vitals

Q: What are Core Web Vitals?

Google's metrics for page experience: (1) LCP (Largest Contentful Paint): how fast the main content loads — target <2.5s. Affected by: slow server, render-blocking resources, large images. (2) INP (Interaction to Next Paint, replaced FID): how fast the page responds to interactions — target <200ms. (3) CLS (Cumulative Layout Shift): how much content moves unexpectedly — target <0.1. Caused by: images without dimensions, dynamic content insertion, web fonts. All three directly affect Google search ranking.

Q: How do you optimize images for the web?

Format: use AVIF (best compression) → WebP → JPEG/PNG via <picture>. Size: serve appropriately sized images using srcset — don't serve 1200px images to mobile. Compression: lossy for photos, lossless for graphics with text/transparency. Loading: lazy load below-fold images (loading="lazy"), fetchpriority="high" for LCP image. Dimensions: always set width and height to prevent CLS. CDN: serve from CDN for caching and edge delivery.

Q: What is CLS (Cumulative Layout Shift) and how do you fix it?

CLS measures unexpected layout shifts — content jumping around as page loads. Causes and fixes: (1) Images without dimensions → always set width/height attributes. (2) Ads/embeds — reserve space with min-height. (3) Web fonts causing FOUT → use font-display: optional or size-adjust to match fallback metrics. (4) Dynamically injected content above existing content → insert below or use fixed dimensions. (5) CSS animations using top/left → use transform instead.

Accessibility

Q: What is the difference between aria-label, aria-labelledby, and aria-describedby?

aria-label: provides a text label directly (string) — for elements with no visible text (icon buttons). aria-labelledby: references another element's text as the label (ID of visible label) — better than aria-label when there's visible text. aria-describedby: additional descriptive text beyond the label (hint, error message, instructions) — read after the label. Priority: native <label for> > aria-labelledby > aria-label. describedby supplements, doesn't replace labeling.

Q: How do you make a modal dialog accessible?

When modal opens: (1) Set aria-modal="true" on dialog element. (2) Move focus to first focusable element inside modal. (3) Trap Tab key — cycle through focusable elements inside modal only. (4) Close on Escape key. When modal closes: (5) Return focus to the element that triggered the modal. Also: role="dialog" with aria-labelledby pointing to modal title. aria-live="assertive" for urgent messages. Never remove content from DOM while dialog is open.

Q: What is the purpose of the alt attribute on images?

alt provides text alternative for images. Screen readers read alt text for users who can't see images. Also displayed when image fails to load. Empty alt="" (not missing): tells screen reader to skip — for decorative images. Missing alt attribute: screen reader reads filename ("hero-image-2024.jpg") — confusing. Descriptive alt: describe what the image shows and its context. For images of text: alt must match the text. For charts: describe the data, not "chart of sales".

Security & Storage

Q: What is the difference between first-party and third-party cookies?

First-party: set by the domain you're visiting (example.com sets cookie for example.com). Used for: session management, preferences, auth. Third-party: set by a different domain (ad.tracker.com sets cookie on example.com) — used for cross-site tracking, ads. Modern browsers increasingly block third-party cookies (Safari, Firefox block by default, Chrome phasing out). SameSite=Strict/Lax cookie attribute prevents cross-site requests from including the cookie.

Q: What is CORS and how does it relate to HTML?

CORS (Cross-Origin Resource Sharing): browser security prevents JS from requesting resources from different origins. Browser sends preflight OPTIONS request, server responds with Access-Control-Allow-Origin header. HTML elements bypass CORS by default: <img src="cross-origin">, <script src="cdn">, <link href="cdn.css">. Adding crossorigin attribute opts into CORS for credentials/SRI. fetch() and XMLHttpRequest are subject to CORS. Solution is server-side headers, not HTML.

Q: How do you implement a Progressive Web App (PWA) with HTML?

Three requirements: (1) HTTPS. (2) Web App Manifest: <link rel="manifest" href="/manifest.json"> — defines name, icons, start_url, display mode (standalone). (3) Service Worker registered via JS: navigator.serviceWorker.register('/sw.js'). Service Worker intercepts fetch requests for offline caching. Add <meta name="apple-mobile-web-app-capable" content="yes"> for iOS. meta theme-color sets browser chrome color on mobile.

Q: What is the difference between Canvas and SVG for building UI elements?

SVG is the right choice for UI elements — it's DOM-based (CSS styling, event listeners work), vector (sharp at all scales), accessible (text is real text, ARIA works), easy to animate with CSS. Canvas is for: games, image processing, complex real-time visualizations with thousands of objects, pixel manipulation (filters, WebGL). For charts: SVG for up to ~1000 data points. Canvas/WebGL for real-time financial charts with continuous updates.

Q: How do you prevent a form from submitting and validate manually?

e.preventDefault() stops default form submission. Then: (1) Use browser's built-in checkValidity() — returns false if any required/pattern/type validation fails. (2) Access validity object per input: input.validity.valueMissing, patternMismatch, etc. (3) Set custom messages: input.setCustomValidity('Error') then reportValidity(). (4) Or use novalidate on form to disable browser UI and handle everything in JS. Libraries like React Hook Form or Zod give much better DX.
