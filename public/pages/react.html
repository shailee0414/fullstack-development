React Fundamentals

Core

What is React?

React is a JavaScript library (not a framework) for building user interfaces. Created by Facebook (Meta) in 2013. It is declarative, component-based, and uses a virtual DOM for efficient updates.

Interview Gold React is a library, not a framework. Angular is a framework. React only handles the View layer — you choose your own routing, state management, etc.
Q: What is the difference between a library and a framework?

A library (React) = you call it. A framework (Angular) = it calls you (Inversion of Control). React gives you tools; you decide the architecture.

Component-Based Architecture

React splits UI into reusable, self-contained pieces called components. Each component manages its own state and renders its own UI. Components can be composed together to build complex UIs.

Component Composition
// Each piece is a self-contained component
function Avatar({ src, name }) {
  return <img src={src} alt={name} />;
}

function UserCard({ user }) {
  return (
    <div className="card">
      <Avatar src={user.avatar} name={user.name} />
      <h2>{user.name}</h2>
    </div>
  );
}

// App composes smaller components
function App() {
  return <UserCard user={{ name: 'Shailesh', avatar: '/pic.jpg' }} />;
}

JSX Basics

JSX is JavaScript XML — a syntax extension that lets you write HTML-like code in JS. Babel transpiles JSX to React.createElement() calls.

JSX vs React.createElement
// JSX (what you write)
const el = <h1 className="title">Hello</h1>;

// What Babel compiles it to
const el = React.createElement('h1', { className: 'title' }, 'Hello');

// JSX Rules:
// 1. Return one root element (or <Fragment>)
// 2. Use className not class
// 3. Self-close empty tags: <img />
// 4. JS expressions in {}
// 5. Attributes are camelCase (onClick, htmlFor)
Common Interview Trap Why can't we return two sibling elements without a wrapper? Because JSX compiles to a single function call (React.createElement) which can only return one value.

Functional Components

Modern React uses function components exclusively. They are plain JavaScript functions that accept props and return JSX. With hooks (React 16.8+), they replaced class components entirely.

Function Component
// Simple function component
function Greeting({ name = 'World' }) {
  return <h1>Hello, {name}!</h1>;
}

// Arrow function style (same thing)
const Greeting = ({ name = 'World' }) => (
  <h1>Hello, {name}!</h1>
);

function Counter({ initialCount }) { // initialCount = prop
  const [count, setCount] = React.useState(initialCount); // count = state

  return (
    <div>

// ❌ Never mutate props

Event Handling

React Event Handling
function Form() {
  const handleSubmit = (e) => {
    e.preventDefault(); // stop page reload
    console.log('submitted');
  };

  const handleChange = (e) => {
    console.log(e.target.value); // get input value
  };

  return (
    <form onSubmit={handleSubmit}>
      <input onChange={handleChange} />
      <button type="submit">Submit</button>
    </form>
  );
}
Note React uses Synthetic Events — cross-browser wrappers over native browser events. They are pooled in React 16 (not React 17+). Always use e.preventDefault() inside React, not return false.

  • Mounting — Component created and inserted into DOM
  • Updating — Re-render due to state or prop change
  • Unmounting — Component removed from DOM

Lifecycle with Hooks — Full Example

Lifecycle via useEffect
function DataLoader({ userId }) {
  const [data, setData] = useState(null);

  // ✅ componentDidMount — runs once after first render
  useEffect(() => {
    console.log('Component mounted');
    return () => console.log('Component unmounted');
  }, []);

  // ✅ componentDidUpdate — runs when userId changes
  useEffect(() => {
    const controller = new AbortController();

    fetch(`/api/user/${userId}`, { signal: controller.signal })
      .then(r => r.json())
      .then(setData);

    // ✅ componentWillUnmount — cleanup
    return () => controller.abort();
  }, [userId]); // re-run when userId changes

  if (!data) return <p>Loading...</p>;
  return <p>{data.name}</p>;
}
Q: Why does useEffect run after render and not before?

React guarantees the DOM is updated before effects run. This mirrors componentDidMount/Update behavior — you can safely access the DOM. If you need to run something synchronously before paint, use useLayoutEffect.

Core Hooks

Core

useState — The Foundation

Returns a stateful value and a function to update it. When the setter is called, React re-renders the component with the new value.

useState — All Patterns
// 1. Simple state
const [count, setCount] = useState(0);

// 2. Object state (spread to update)
const [user, setUser] = useState({ name: '', age: 0 });
setUser(prev => ({ ...prev, name: 'Shailesh' })); // ✅ functional update

// 3. Lazy initialization (expensive computation runs ONCE)
const [data, setData] = useState(() => heavyCompute());

// 4. Functional updates (ALWAYS use for count-based logic)
// ❌ Wrong — may use stale state
setCount(count + 1);
// ✅ Correct — always gets latest value
setCount(prev => prev + 1);

// 5. Arrays — never mutate, always return new array
const [items, setItems] = useState([]);
setItems(prev => [...prev, newItem]);       // add
setItems(prev => prev.filter(i => i.id !== id)); // remove
Common Mistake State updates are asynchronous! Calling setCount(count+1) three times in a row will NOT increment by 3 — it increments by 1 because count is stale. Use functional form: setCount(c => c+1) three times = increments by 3.
Q: What is "batching" in React state updates?

React batches multiple setState calls in the same event handler into a single re-render for performance. React 18 introduced automatic batching — even async code (setTimeout, fetch callbacks) is batched. Use flushSync() to opt out.

useEffect — Deep Dive

Runs side effects after render. The dependency array controls WHEN it runs.

useEffect Dependency Variants
// No deps → runs after EVERY render (usually a bug)
useEffect(() => { console.log('every render'); });

// Empty deps → runs ONCE (componentDidMount)
useEffect(() => { fetchData(); }, []);

// With deps → runs when deps change
useEffect(() => { fetchUser(id); }, [id]);

// Cleanup (componentWillUnmount)
useEffect(() => {
  const timer = setInterval(() => tick(), 1000);
  return () => clearInterval(timer); // ✅ cleanup
}, []);
Why async is NOT allowed directly in useEffect
// ❌ WRONG — async function returns a Promise, not a cleanup fn
useEffect(async () => {
  const data = await fetch('/api');
  // React expects cleanup fn OR undefined, not a Promise
}, []);

// ✅ CORRECT — define async inside
useEffect(() => {
  let cancelled = false;

  const load = async () => {
    const res = await fetch('/api');
    const data = await res.json();
    if (!cancelled) setData(data); // guard against stale update
  };

  load();
  return () => { cancelled = true; }; // cleanup
}, []);
Q: What is a race condition in useEffect and how to fix it?

If userId changes rapidly, multiple fetches fire. The slower one might resolve last with stale data. Fix: use a cancelled flag or AbortController. The cleanup function runs before the next effect, so set cancelled=true or call controller.abort() there.

useContext — Consume Context

useContext Pattern
const ThemeContext = createContext('light');

function App() {
  return (
    <ThemeContext.Provider value="dark">
      <Toolbar />
    </ThemeContext.Provider>
  );
}

function Toolbar() {
  const theme = useContext(ThemeContext); // 'dark'
  return <button className={theme}>Click</button>;
}
Performance Warning Every component using useContext re-renders when context value changes — even if they only use part of the value. Fix: split context into smaller contexts, use memo, or use a state manager like Zustand.

useRef — Two Use Cases

useRef — DOM Access & Persistent Values
function TextInput() {
  const inputRef = useRef(null);

  // Use Case 1: DOM access
  const focus = () => inputRef.current.focus();

  return <>
    <input ref={inputRef} />
    <button onClick={focus}>Focus</button>
  </>;
}

function Timer() {
  const [count, setCount] = useState(0);
  const timerRef = useRef(null);

  // Use Case 2: Mutable value that persists WITHOUT re-render
  // Changing ref.current does NOT cause re-render
  const start = () => {
    timerRef.current = setInterval(() => {
      setCount(c => c + 1);
    }, 1000);
  };

  const stop = () => clearInterval(timerRef.current);

  return <div>{count} <button onClick={start}>Start</button></div>;
}
Q: Difference between useRef and useState?

useState — changing value triggers re-render. useRef — changing .current does NOT re-render. Use ref for values you need to persist across renders but don't need to trigger UI updates (timers, previous values, DOM refs).

useRef to track previous value (classic interview pattern)
function usePrevious(value) {
  const ref = useRef();
  useEffect(() => {
    ref.current = value; // runs after render, so stores prev
  });
  return ref.current; // returns value from previous render
}

function Counter() {
  const [count, setCount] = useState(0);
  const prev = usePrevious(count);
  return <p>Now: {count}, Before: {prev}</p>;
}

useReducer — Complex State Logic

useReducer is preferred over useState when state logic is complex or next state depends on previous state. Think of it as a mini Redux.

useReducer — Shopping Cart Example
const initialState = { items: [], total: 0 };

function cartReducer(state, action) {
  switch (action.type) {
    case 'ADD_ITEM':
      return {
        ...state,
        items: [...state.items, action.payload],
        total: state.total + action.payload.price
      };
    case 'REMOVE_ITEM':
      return {
        ...state,
        items: state.items.filter(i => i.id !== action.payload),
        total: state.total - state.items.find(i => i.id === action.payload).price
      };
    case 'CLEAR':
      return initialState;
    default:
      return state; // always return state for unknown actions
  }
}

function Cart() {
  const [state, dispatch] = useReducer(cartReducer, initialState);

  return (
    <div>
      <p>Total: ₹{state.total}</p>
      <button onClick={() => dispatch({ type: 'CLEAR' })}>Clear</button>
    </div>
  );
}
When to use useReducer vs useState? Use useReducer when: multiple related state values, next state depends on previous, state logic is complex enough to test separately, or you need predictable state transitions (like a state machine).

Performance Hooks

Performance

useMemo — Memoize Computed Values

Caches the result of an expensive calculation. Re-computes only when dependencies change.

useMemo — Correct Usage
function ProductList({ products, searchTerm }) {
  // ✅ Expensive filter — only recomputes when inputs change
  const filtered = useMemo(
    () => products.filter(p => p.name.includes(searchTerm)),
    [products, searchTerm]
  );

  // ❌ Don't memoize cheap operations
  const doubled = useMemo(() => count * 2, [count]); // overkill
  const doubled2 = count * 2; // ✅ just compute it

  return <ul>{filtered.map(p => <li key={p.id}>{p.name}</li>)}</ul>;
}

useCallback — Memoize Functions

Caches a function reference. Essential when passing callbacks to memoized child components (React.memo) to prevent unnecessary re-renders.

useCallback — The Key Use Case
// Without useCallback, new function reference every render
// → MemoizedChild re-renders even if nothing changed
function Parent() {
  const [count, setCount] = useState(0);
  const [name, setName] = useState('');

  // ✅ Stable reference — only changes if setCount changes (never)
  const increment = useCallback(() => {
    setCount(c => c + 1);
  }, []); // empty deps: setCount is stable

  return (
    <div>
      <input value={name} onChange={e => setName(e.target.value)} />
      // MemoizedButton won't re-render when name changes
      <MemoizedButton onClick={increment} />
    </div>
  );
}

const MemoizedButton = React.memo(({ onClick }) => {
  console.log('Button rendered'); // only logs once!
  return <button onClick={onClick}>+</button>;
});
Interview Trap useCallback is ONLY useful when the child is wrapped in React.memo. Without memo, useCallback just adds overhead. useMemo returns a value. useCallback returns a function (it's essentially useMemo(() => fn, deps)).
Q: What is the difference between useMemo and useCallback?

useMemo caches a computed value: useMemo(() => expensiveCalc(), [dep]). useCallback caches a function reference: useCallback(() => doThing(), [dep]). useCallback(fn, deps) is exactly equivalent to useMemo(() => fn, deps).

Advanced Hooks

Advanced

useLayoutEffect — Synchronous DOM Measurement

Like useEffect but fires synchronously before browser paint. Use for DOM measurements and animations that would flicker with useEffect.

useLayoutEffect vs useEffect
// Order of operations:
// 1. React renders (reconciliation)
// 2. React commits DOM changes
// 3. useLayoutEffect runs (SYNC — blocks paint)
// 4. Browser PAINTS
// 5. useEffect runs (ASYNC — after paint)

function Tooltip({ children }) {
  const ref = useRef();
  const [width, setWidth] = useState(0);

  // ✅ Measure before paint to avoid flicker
  useLayoutEffect(() => {
    setWidth(ref.current.getBoundingClientRect().width);
  }, []);

  return <div ref={ref}>{children} ({width}px)</div>;
}

useImperativeHandle — Expose Ref API

Used with forwardRef to customize the ref handle exposed to parent. Lets parent call methods on child component.

useImperativeHandle + forwardRef
const FancyInput = React.forwardRef((props, ref) => {
  const inputRef = useRef();

  // Expose only specific methods to parent
  useImperativeHandle(ref, () => ({
    focus: () => inputRef.current.focus(),
    clear: () => { inputRef.current.value = ''; },
  }));

  return <input ref={inputRef} {...props} />;
});

function Parent() {
  const ref = useRef();
  return (
    <div>
      <FancyInput ref={ref} />
      <button onClick={() => ref.current.focus()}>Focus</button>
      <button onClick={() => ref.current.clear()}>Clear</button>
    </div>
  );
}

useDebugValue — DevTools Label

useDebugValue in custom hooks
function useOnlineStatus() {
  const [isOnline, setIsOnline] = useState(navigator.onLine);

  // Shows "OnlineStatus: Online" in React DevTools
  useDebugValue(isOnline ? 'Online' : 'Offline');

  useEffect(() => {
    const on = () => setIsOnline(true);
    const off = () => setIsOnline(false);
    window.addEventListener('online', on);
    window.addEventListener('offline', off);
    return () => {
      window.removeEventListener('online', on);
      window.removeEventListener('offline', off);
    };
  }, []);

  return isOnline;
}

React 18 Hooks

React 18

useTransition — Non-Blocking UI Updates

Marks updates as non-urgent. React will interrupt them if more urgent updates come in. Perfect for search, filtering, tab switching.

useTransition — Search Example
function SearchPage({ items }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(items);
  const [isPending, startTransition] = useTransition();

  const handleSearch = (e) => {
    const val = e.target.value;
    setQuery(val); // ✅ urgent — input must be responsive

    startTransition(() => {
      // ⚡ non-urgent — can be interrupted
      setResults(items.filter(i => i.includes(val)));
    });
  };

  return (
    <div>
      <input value={query} onChange={handleSearch} />
      {isPending && <span>Updating...</span>}
      <ResultsList items={results} />
    </div>
  );
}
Q: What is Concurrent Mode / Concurrent Features in React 18?

React 18's concurrent features allow React to interrupt, pause, and resume renders. useTransition and useDeferredValue are the main hooks for opting into this behavior. It prevents expensive renders from blocking urgent UI updates (like typing in an input).

useDeferredValue — Defer Slow Renders

Like useTransition but for values you receive (not set). Useful when you can't access the state setter (e.g., props from parent).

useDeferredValue
function SearchResults({ query }) {
  // Deferred version lags behind query — allows input to stay snappy
  const deferredQuery = useDeferredValue(query);

  // This expensive component renders with the deferred (possibly stale) value
  const results = useMemo(
    () => heavyFilter(deferredQuery),
    [deferredQuery]
  );

  const isStale = deferredQuery !== query;

  return (
    <div style={{ opacity: isStale ? 0.7 : 1 }}>
      {results.map(r => <div key={r.id}>{r.name}</div>)}
    </div>
  );
}

// Difference: useTransition — you control the setter
// useDeferredValue — you only have the value (e.g., from props)

function useWindowWidth() {

Shows optimistic (assumed successful) state immediately while the async operation completes. Automatically rolls back if it fails.

useOptimistic — Like Button
function LikeButton({ postId, initialLikes }) {
  const [likes, setLikes] = useState(initialLikes);
  const [optimisticLikes, setOptimisticLikes] = useOptimistic(likes);

  const handleLike = async () => {
    setOptimisticLikes(l => l + 1); // shows +1 instantly

    try {
      const updated = await likePost(postId);
      setLikes(updated.likes); // sync with server
    } catch {
      // React auto-reverts optimisticLikes to actual likes
    }
  };

  return <button onClick={handleLike}>❤ {optimisticLikes}</button>;
}

use() Hook — Suspend on Promises

The use() hook can unwrap Promises and Context inside render — including inside conditionals and loops (unlike other hooks).

use() Hook
// With Suspense — component suspends while promise resolves
function UserProfile({ userPromise }) {
  const user = use(userPromise); // suspends here if pending
  return <h1>{user.name}</h1>;
}

// Can be used conditionally (unlike other hooks!)
function App({ showUser, userPromise }) {
  if (showUser) {
    const user = use(userPromise); // ✅ valid!
  }
}

// Wrap in Suspense
<Suspense fallback={<Spinner />}>

useEffectEvent (experimental) — Stable Event Handlers

Solves the problem of event handlers needing to read latest values without being in the dependency array.

useEffectEvent
function ChatRoom({ roomId, onMessage }) {

  }, [roomId]); // onMessage not in deps — no stale closure issue
}

Rendering, Virtual DOM & Reconciliation

🔥 Most Asked

What is the Virtual DOM?

The Virtual DOM (VDOM) is a lightweight in-memory JavaScript representation of the real DOM tree. React maintains two VDOM trees: the current one and the next one (after state change). It diffs them and only updates the real DOM with the minimum changes needed.

Interview Answer Template "React maintains a virtual DOM — a JS object tree mirroring the real DOM. On state change, it creates a new VDOM, diffs it against the previous via reconciliation, and applies only the minimal set of real DOM changes. This is more efficient than directly manipulating the DOM every time."
VDOM Mental Model
// Real DOM (expensive to manipulate):
// <div><h1>Hello</h1></div>

// Virtual DOM (cheap JS object):
{
  type: 'div',
  props: {},
  children: [{
    type: 'h1',
    props: {},
    children: ['Hello']
  }]
}

// After state change, React creates a NEW vdom tree
// Runs the diff algorithm (reconciliation)
// Computes minimal DOM operations needed
// Batches and applies them to real DOM

Reconciliation Algorithm

React's algorithm for comparing two VDOM trees (diffing). It has two main heuristics:

  • Different element type → tear down old tree, build new one from scratch
  • Same element type → keep DOM node, only update changed attributes
  • Lists → use key prop to identify which items moved/added/removed
Reconciliation Examples
// Case 1: Different type → full remount
// Before: <div><Counter /></div>
// After:  <span><Counter /></span>
// Result: Counter unmounts and remounts (state LOST)

// Case 2: Same type → update in place
// Before: <div className="a" />
// After:  <div className="b" />
// Result: React updates className only (state PRESERVED)

// Case 3: Lists without key (BAD)
['a', 'b', 'c'] => ['x', 'a', 'b', 'c'] // React mutates all 4

// Case 4: Lists with key (GOOD)
// React knows 'a','b','c' are unchanged — only inserts 'x'
items.map(item => <li key={item.id}>{item.name}</li>)
Key Deep Dive — Most Common Interview Question Keys must be stable, unique, and predictable. Using array index as key causes bugs when list order changes — React gets confused about which item is which. Always use a unique ID from your data.
Q: Why is using index as key bad?

If you insert an item at position 0, all other items shift. React sees the key=0 component as the same component but with different props, causing wrong state retention and potential rendering bugs. E.g., form inputs retain values of the wrong item.

Q: Can two siblings have the same key?

Keys only need to be unique among siblings — not globally. But duplicate keys among siblings will cause React to skip rendering one of them.

React Fiber — The Reconciler

Fiber is React's reimplemented reconciliation algorithm (React 16+). It enables incremental rendering — breaking work into chunks that can be paused, aborted, or prioritized. This is what powers Concurrent Mode.

Old Stack ReconcilerFiber Reconciler
Synchronous, blockingInterruptible, async capable
Once started, must finishCan pause and resume
No priority levelsWork has priority (lane model)
No Concurrent ModeEnables useTransition, Suspense

React Render vs Commit Phases

  • Render Phase (pure) — React calls components, creates VDOM, runs reconciliation. Can be interrupted. No side effects here.
  • Commit Phase (impure) — React applies changes to real DOM. Synchronous, cannot be interrupted. useLayoutEffect runs here.
StrictMode Double Render In development + StrictMode, React renders components twice (render phase only) to detect side effects. This is why you see console.logs twice. It does NOT double-commit — DOM updates happen once.

API & Async Concepts

Advanced

AbortController — Cancel Requests

AbortController Pattern
useEffect(() => {
  const controller = new AbortController();

  const load = async () => {
    try {
      const res = await fetch(`/api/users/${id}`, {
        signal: controller.signal // link to controller
      });
      const data = await res.json();
      setUser(data);
    } catch (err) {
      if (err.name === 'AbortError') return; // expected — ignore
      setError(err);
    }
  };

  load();
  return () => controller.abort(); // cancels on cleanup
}, [id]);

Custom Data Fetching Hook

useFetch — reusable pattern
function useFetch(url) {
  const [state, dispatch] = useReducer(
    (s, action) => ({ ...s, ...action }),
    { data: null, loading: true, error: null }
  );

  useEffect(() => {
    if (!url) return;
    const ctrl = new AbortController();

    dispatch({ loading: true, error: null });

    fetch(url, { signal: ctrl.signal })
      .then(r => r.json())
      .then(data => dispatch({ data, loading: false }))
      .catch(err => {
        if (err.name !== 'AbortError')
          dispatch({ error: err.message, loading: false });
      });

    return () => ctrl.abort();
  }, [url]);

  return state;
}

// Usage
const { data, loading, error } = useFetch(`/api/users/${id}`);

Debounce / Throttle in React

function useDebounce(value, delay = 300) {
  const [debounced, setDebounced] = useState(value);

  useEffect(() => {
    const timer = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(timer); // cancel if value changes fast
  }, [value, delay]);

  return debounced;
}

  useEffect(() => {

  }, [debouncedQuery]);

  return <input value={query} onChange={e => setQuery(e.target.value)} />;
}

Higher Order Components (HOC)

A HOC is a function that takes a component and returns a new, enhanced component. Pattern from class component era — largely replaced by hooks but still asked in interviews.

HOC Pattern
// HOC — adds authentication check
function withAuth(WrappedComponent) {
  return function AuthComponent(props) {
    const isLoggedIn = useAuth();

    if (!isLoggedIn) return <Redirect to="/login" />;
    return <WrappedComponent {...props} />;
  };
}

// HOC — adds loading state
function withLoading(Component) {
  return ({ isLoading, ...props }) =>
    isLoading ? <Spinner /> : <Component {...props} />;
}

Q: HOC vs Custom Hooks — which to use?

Custom Hooks are preferred for logic reuse (no wrapper component, no prop conflicts, better DevTools readability). HOCs are useful for rendering logic (conditional rendering, wrapping UI). HOCs can cause "wrapper hell" with multiple HOCs stacked.

Custom Hooks — Reusable Logic

// useLocalStorage
function useLocalStorage(key, initialValue) {
  const [value, setValue] = useState(() => {
    try {
      const stored = localStorage.getItem(key);
      return stored ? JSON.parse(stored) : initialValue;
    } catch { return initialValue; }
  });

  const set = useCallback((val) => {
    setValue(val);
    localStorage.setItem(key, JSON.stringify(val));
  }, [key]);

  return [value, set];
}

// useToggle
function useToggle(initial = false) {
  const [value, setValue] = useState(initial);
  const toggle = useCallback(() => setValue(v => !v), []);
  return [value, toggle];
}

// useClickOutside
function useClickOutside(ref, callback) {
  useEffect(() => {
    const handler = (e) => {
      if (ref.current && !ref.current.contains(e.target)) {
        callback();
      }
    };
    document.addEventListener('mousedown', handler);
    return () => document.removeEventListener('mousedown', handler);
  }, [ref, callback]);
}

Full Context Pattern with Performance

Context + Reducer + Split Pattern
// Split context into state and dispatch to prevent re-renders
const StateContext = createContext();
const DispatchContext = createContext();

const initialState = { user: null, theme: 'dark' };

function appReducer(state, action) {
  switch(action.type) {
    case 'SET_USER': return { ...state, user: action.payload };
    case 'SET_THEME': return { ...state, theme: action.payload };
    default: return state;
  }
}

export function AppProvider({ children }) {
  const [state, dispatch] = useReducer(appReducer, initialState);

  return (
    <StateContext.Provider value={state}>
      <DispatchContext.Provider value={dispatch}>
        {children}
      </DispatchContext.Provider>
    </StateContext.Provider>
  );
}

// Custom hooks for consumers
export const useAppState = () => useContext(StateContext);
export const useAppDispatch = () => useContext(DispatchContext);

// Components that only dispatch won't re-render on state change
function LogoutButton() {
  const dispatch = useAppDispatch(); // only subscribes to dispatch
  return <button onClick={() => dispatch({ type: 'SET_USER', payload: null })}>Logout</button>;
}
Q: When should you use Context vs Redux/Zustand?

Context: small apps, infrequent updates (theme, auth, locale). Redux/Zustand: frequent updates, complex state, need DevTools, time-travel debugging, or performance-critical state shared widely. Context re-renders all consumers; external stores can subscribe selectively.

Error Boundaries

Core

What Are Error Boundaries?

Error boundaries are class components that catch JavaScript errors anywhere in their child component tree and display a fallback UI instead of crashing the whole app. They catch errors during rendering, in lifecycle methods, and in constructors of child components.

Limitation Error boundaries do NOT catch: event handler errors (use try/catch), async errors (use try/catch or react-error-boundary), server-side rendering errors, or errors thrown in the error boundary itself.
Error Boundary Implementation
class ErrorBoundary extends React.Component {
  state = { hasError: false, error: null };

  static getDerivedStateFromError(error) {
    // Update state to show fallback UI
    return { hasError: true, error };
  }

  componentDidCatch(error, info) {
    // Log to error reporting service
    logErrorToService(error, info.componentStack);
  }

  render() {
    if (this.state.hasError) {
      return (
        <div>
          <h2>Something went wrong</h2>
          <button onClick={() => this.setState({ hasError: false })}>
            Try Again
          </button>
        </div>
      );
    }
    return this.props.children;
  }
}

Forms — Controlled vs Uncontrolled

Core

Controlled Components

React state is the single source of truth. Every input has a value and onChange handler. React controls the input.

Controlled Form
function SignupForm() {
  const [form, setForm] = useState({ email: '', password: '' });
  const [errors, setErrors] = useState({});

  const handleChange = ({ target: { name, value } }) => {
    setForm(prev => ({ ...prev, [name]: value }));
  };

  const validate = () => {
    const errs = {};
    if (!form.email.includes('@')) errs.email = 'Invalid email';
    if (form.password.length < 8) errs.password = 'Too short';
    return errs;
  };

  const handleSubmit = (e) => {
    e.preventDefault();
    const errs = validate();
    if (Object.keys(errs).length) { setErrors(errs); return; }
    submitForm(form);
  };

  return (
    <form onSubmit={handleSubmit}>
      <input name="email" value={form.email} onChange={handleChange} />
      {errors.email && <span>{errors.email}</span>}
      <input name="password" type="password"
             value={form.password} onChange={handleChange} />
      <button type="submit">Sign Up</button>
    </form>
  );
}

function TodoList() {
  const [todos, setTodos] = useState([]);

  const toggleTodo = async (id) => {
    // 1. Save previous state for rollback
    const prev = todos;

    // 2. Optimistically update UI
    setTodos(todos => todos.map(t =>
      t.id === id ? { ...t, done: !t.done } : t
    ));

    try {
      // 3. Attempt server update
      await api.toggleTodo(id);
    } catch (err) {
      // 4. Rollback on failure
      setTodos(prev);
      toast.error('Failed to update. Please retry.');
    }
  };

  return todos.map(todo => (
    <TodoItem key={todo.id} todo={todo} onToggle={toggleTodo} />
  ));
}

Lazy Loading Images

Intersection Observer Hook
function useLazyLoad(threshold = 0.1) {
  const ref = useRef();
  const [visible, setVisible] = useState(false);

  useEffect(() => {
    const observer = new IntersectionObserver(
      ([entry]) => entry.isIntersecting && setVisible(true),
      { threshold }
    );
    if (ref.current) observer.observe(ref.current);
    return () => observer.disconnect();
  }, [threshold]);

  return [ref, visible];
}

function LazyImage({ src, alt }) {
  const [ref, visible] = useLazyLoad();
  return (
    <img ref={ref} src={visible ? src : ''} alt={alt} />
  );
}

Common Interview Questions Rapid Fire

Q: What happens when you call setState inside render?

Infinite loop! setState triggers re-render, render calls setState again. Always call state setters in event handlers or effects, never directly in render body (except useReducer dispatch for lazy state init).

Q: How does React handle multiple setState calls in same event handler?

React 18 batches all setState calls in event handlers AND async code into a single re-render. In React 17, only event handler calls were batched. Use flushSync() to force immediate sync re-render (e.g., before reading DOM).

Q: What is prop drilling and how to avoid it?

Prop drilling: passing props through many intermediate components that don't use them. Solutions: Context API (for global state), component composition (pass component, not data), state management (Redux/Zustand).

Q: What is the children prop?

Everything between opening and closing tags of a component is accessible via props.children. Can be a string, element, array, or function (render props pattern). Used for reusable container components like Card, Modal, Layout.

Q: Explain React's one-way data flow

Data flows down (parent → child via props). Events flow up (child → parent via callback props). This makes data flow predictable and easier to debug. Two-way binding requires explicit onChange handlers (controlled components).

Q: What is Strict Mode and why does it double-render?

Strict Mode double-invokes render functions and effects in development to help detect side effects in impure components. If your component behaves differently on second render, you have a purity bug. No production impact.

Q: What is hydration in React?

In Server-Side Rendering (SSR), the server sends HTML. The browser displays it immediately (fast first paint). Then React "hydrates" — attaches event listeners and takes over. Mismatch between server and client HTML = hydration error.

Q: Difference between controlled and uncontrolled inputs?

Controlled: React state = single source of truth, value prop + onChange required. Uncontrolled: DOM manages input state, read via ref. Controlled is preferred as it enables instant validation, conditional disabling, and format enforcement.
